# Task #24 수행계획서

GitHub Issue: [#24](https://github.com/postmelee/crop/issues/24)
마일스톤: M020

## 목적

요소 선택 모드에서 너무 큰 wrapper가 자동 선택 후보로 추천되는 문제를 고친다. 사용자가 NamuWiki 유사 레이아웃의 큰 페이지 wrapper 위를 지나갈 때 viewport 크기로 잘린 큰 빈 영역이 선택 박스로 확정되지 않아야 한다.

이번 task의 결과는 자동 선택 후보 정책의 보정이다. 큰 wrapper 내부의 실제 콘텐츠, 표, infobox, 카드처럼 사용자가 의도하기 쉬운 적정 크기 요소는 계속 자동 선택 가능해야 하며, Copy/Save는 잘못된 wrapper fallback rect가 아니라 실제 선택 rect 기준으로 동작해야 한다.

## 배경

이슈 #24는 NamuWiki `턴스타일` 페이지에서 큰 wrapper가 선택 후보로 잡히고, 선택 박스와 저장 결과가 큰 빈 영역 위주로 잘리는 문제를 보고한다. Firefox Screenshots는 같은 상황에서 `MAX_DETECT_WIDTH`, `MAX_DETECT_HEIGHT` 기준으로 너무 큰 자동 선택 영역을 무시하거나 직전의 유효한 더 작은 후보를 유지하는 흐름을 사용한다.

현재 `src/firefox-derived/overlay-helpers.ts`의 `getBestRectForElement()`는 초기 target 자체가 너무 클 때 `getFallbackRect()`로 viewport와 교차한 rect를 만들고, 그 rect가 다시 threshold보다 크면 `maxDetectWidth`/`maxDetectHeight` 크기로 잘라 자동 선택 후보로 반환할 수 있다. 이 fallback은 사용자가 실제로 의도하지 않은 큰 wrapper를 선택 후보로 확정시키는 원인이 될 수 있다.

## 범위

### 포함

- `src/firefox-derived/overlay-helpers.ts`의 자동 선택 후보 산정 로직 분석 및 수정
- 너무 큰 요소에 대한 viewport 기반 fallback rect 제거 또는 제한
- 직전 유효 후보가 있을 때 큰 parent/wrapper를 만나면 해당 후보를 유지하는 정책 고정
- 큰 wrapper, 내부 table/infobox, 일반 카드 선택 시나리오 단위 테스트 추가/수정
- Phase 6 품질 매트릭스 또는 smoke 문서에 "너무 큰 wrapper는 자동 추천하지 않음" 정책 반영
- Copy/Save가 선택 rect 기준으로 유지되는지 기존 crop 경로 회귀 검증

### 제외

- 수동 드래그 선택 정책 변경
- 전체 페이지 캡처, visible viewport 캡처 팝업 UI 변경
- Chrome 권한 변경
- `debugger`, `<all_urls>` 또는 broad host permission 추가
- Firefox 제품명/브랜딩 사용
- Firefox privileged API 직접 포팅
- cross-origin iframe 내부 DOM 접근 정책 변경

## 설계 방향

- `getBestRectForElement()`는 큰 요소를 자동 선택 후보로 만들기 위해 viewport 크기로 잘라 반환하지 않는다. 큰 요소를 만나기 전 더 작은 유효 후보가 있으면 그 후보를 유지하고, 없으면 `null`을 반환하는 방향을 기본값으로 한다.
- `MAX_DETECT_WIDTH`/`MAX_DETECT_HEIGHT` 계열 threshold는 "잘라낼 최대 크기"가 아니라 "자동 선택 후보 제외 기준"으로 사용한다.
- 너무 작은 child에서 parent로 승격하는 기존 흐름은 유지하되, 승격 대상 parent가 너무 크면 직전의 유효 child 후보 또는 sibling 확장 결과를 우선한다.
- `role="article"` parent 승격, heading 자체 선택 회피, same-origin iframe rect projection, open shadow traversal은 기존 회귀 테스트를 유지한다.
- table/infobox처럼 viewport 밖까지 확장될 수 있지만 의미 있는 콘텐츠 요소는 raw rect가 threshold 내에 있거나 직전 후보로 유지되는 경우 정상 선택되게 한다.
- Copy/Save backend는 이미 선택 rect를 입력으로 받으므로 이번 task에서는 capture backend를 바꾸지 않는다. 필요한 경우 회귀 테스트나 smoke 기준만 보강한다.
- Firefox source는 동작 기준으로 참고하되, 새 MPL 코드 복사가 필요해지면 구현을 중단하고 별도 승인받는다.

## 잠정 단계

- **Stage 1 — 자동 후보 정책과 회귀 테스트 기준 고정**
  - 기존 `getBestRectForElement()`의 큰 요소 fallback 동작을 테스트로 재현한다.
  - "큰 wrapper 단독 hover는 후보 없음", "큰 parent 진입 전 유효 child 후보 유지", "table/infobox 후보 유지" 기대값을 단위 테스트로 고정한다.
  - iframe fallback과 role article 승격 테스트가 영향을 받지 않는지 확인한다.

- **Stage 2 — 큰 wrapper fallback 제거 또는 제한 구현**
  - `overlay-helpers.ts`에서 너무 큰 요소를 viewport `maxDetect` 크기로 잘라 반환하는 경로를 제거하거나 제한한다.
  - threshold override와 기존 `setMaxDetectWidth()`/`setMaxDetectHeight()` 테스트 의미를 새 정책에 맞게 수정한다.
  - sibling 확장과 article parent 승격이 큰 wrapper를 다시 후보로 만들지 않도록 확인한다.

- **Stage 3 — fixture/smoke와 Copy/Save 회귀 보강**
  - 필요 시 Phase 6 fixture에 NamuWiki 유사 큰 wrapper, 내부 infobox/table/card target을 추가한다.
  - overlay runtime이 `null` 후보를 받았을 때 이전 highlight를 부적절하게 유지하지 않는지 확인한다.
  - Copy/Save 경로가 선택 rect와 일치하는지 기존 crop-image/phase6 회귀 테스트로 확인한다.

- **Stage 4 — 문서, 권한 경계, 최종 검증과 보고**
  - 품질 매트릭스에 큰 wrapper 자동 추천 제외 항목과 검증 근거를 반영한다.
  - `manifest.json` 권한이 변경되지 않았음을 확인한다.
  - 최종 보고서와 PR 요약에 Firefox parity 판단과 남은 제한을 정리한다.

## 리스크

- **선택 후보 없음 상태의 UX 회귀**: 큰 wrapper 단독 hover에서 `null`을 반환하면 overlay가 이전 highlight를 유지하거나 깜빡일 수 있다. overlay runtime의 null 처리와 fixture smoke로 확인한다.
- **너무 작은 child와 큰 parent 사이의 경계**: 작은 child를 parent로 승격하는 기존 휴리스틱이 큰 wrapper 제거 정책과 충돌할 수 있다. 직전 유효 후보 유지와 absolute minimum reject 기준을 테스트로 나눠 고정한다.
- **table/infobox 오탐 방지**: 의미 있는 table이 threshold를 초과할 경우 후보에서 빠질 수 있다. 이슈 수용 기준에 맞춰 NamuWiki 유사 infobox/table fixture를 별도 검증하고 필요 시 threshold 또는 후보 유지 규칙을 조정한다.
- **Firefox parity 과대 해석**: Firefox 원본 privileged API와 전체 구조를 그대로 포팅하지 않는다. 이번 task는 자동 선택 후보 threshold 정책만 맞춘다.
- **권한 범위 확대 유혹**: 실제 웹 smoke 편의를 위해 host permission을 넓히지 않는다. 필요하면 별도 task로 분리한다.
