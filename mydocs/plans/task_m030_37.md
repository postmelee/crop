# Task #37 수행계획서 - Chrome Web Store Developer Dashboard 입력값 확정 및 제출 체크리스트

GitHub Issue: [#37](https://github.com/postmelee/crop/issues/37)
마일스톤: M030

## 목적

Chrome Web Store 실제 제출 직전에 Developer Dashboard에 입력할 값을 확정한다. Store Listing, Privacy practices, Distribution, package upload, permission justification, review submit 전 확인 항목을 작업지시자가 그대로 대조할 수 있는 표로 정리한다.

이번 task의 결과는 제출 입력값과 수동 smoke checklist의 최종 문서화다. 실제 Chrome Web Store upload, review submit, 게시 상태 변경은 수행하지 않고 작업지시자 승인 필요 항목으로 남긴다.

## 배경

#9에서 Chrome Web Store 배포 준비 문서, `PRIVACY.md`, 권한/privacy copy, release package checklist, manifest/Store icon asset을 정리했고 PR #36이 `devel`에 merge됐다. #9 최종 보고서에는 Store screenshot, small promotional image, Dashboard 입력값, upload/review submit 승인 항목이 blocker로 남아 있었다.

작업 시작 직전 확인한 최신 PR은 #38 `Task #35: 전체 페이지 캡처 다운스케일 fallback`이다. PR #38은 2026-06-04에 `devel`로 merge됐고, 현재 기준 브랜치 `origin/devel`의 HEAD는 merge commit `0a6c1b7`이다. 이 변경으로 full-page oversized capture 설명은 "명시적 size error" 중심이 아니라 "단일 PNG 유지를 위한 자동 downscale fallback, 단 해상도 저하와 동적 레이아웃 한계는 유지" 기준으로 반영해야 한다.

## 설계 방향

- #37의 최종 입력값은 새 기술 노트 `mydocs/tech/task_m030_37_chrome_web_store_dashboard.md`에 둔다. #9 기술 노트는 선행 준비 산출물로 대조하되, 역사 기록을 직접 덮어쓰지 않는다.
- Dashboard 입력값은 Store Listing, Privacy practices, Distribution, Package/Upload, Review submit 전 확인 항목으로 분리한 표를 기본 구조로 사용한다.
- 입력값은 현재 `manifest.json`의 권한 `activeTab`, `scripting`, `clipboardWrite`, `downloads`와 실제 사용처를 기준으로 작성한다.
- privacy disclosure는 local processing, no server upload, no telemetry/analytics, explicit Copy/Save behavior를 유지한다.
- PR #38 반영에 따라 full-page large canvas 설명은 자동 downscale fallback과 선명도 저하 가능성을 함께 적는다. `PRIVACY.md` 또는 #9 Store copy 초안에 남은 낡은 "explicit size errors" 표현은 필요 시 #37 범위에서 보정한다.
- privacy policy URL은 PR merge 후 stable URL 후보를 제시하되, 실제 Dashboard 입력은 제출 승인 단계로 남긴다.
- Store screenshot과 small promotional image는 이번 task에서 제작하지 않고 필수/선택/후속/승인 필요 항목으로 분리한다.
- 공식 문서 확인일과 URL, 실제 Dashboard 확인 가능 여부를 #37 기술 노트에 기록한다.

## 잠정 단계

- **Stage 1 — 공식 문서와 현행 산출물 재대조**
  - Chrome Web Store 공식 문서와 실제 Dashboard 항목명을 확인하고, #9 산출물과 PR #38 이후 현재 README/PRIVACY/manifest 상태를 대조한다.
  - #37 기술 노트에 확인일, URL, Dashboard 확인 가능 여부, PR #38 반영 차이를 기록한다.
- **Stage 2 — Dashboard 입력값 표 확정**
  - Store Listing, Privacy practices, permission justification, Distribution, URL 후보를 표로 확정한다.
  - full-page downscale fallback, no telemetry/no server, Copy/Save explicit action, 권한 최소화 문구가 서로 충돌하지 않는지 확인한다.
- **Stage 3 — Package/Upload와 제출 전 smoke checklist 확정**
  - `npm run build`, fresh zip 생성, `dist/manifest.json`, zip contents 확인 절차를 실행하고 upload/review submit 전 수동 checklist를 갱신한다.
  - screenshot, small promotional image, optional marquee/video, localized listing을 필수/선택/후속으로 분리한다.
- **Stage 4 — 통합 검증과 최종 보고**
  - build/typecheck/test/grep/diff 검증을 수행하고, 최종 보고서와 오늘할일 상태를 정리한다.
  - 실제 submit은 수행하지 않았음을 명확히 남기고, 작업지시자 승인 필요 항목을 분리한다.

## 리스크

- **Dashboard 항목 변동성**: Chrome Web Store Dashboard UI와 공식 문서는 변경될 수 있다. 공식 URL과 확인일을 남기고, 실제 submit 전 재확인 항목을 분리한다.
- **실제 Dashboard 접근 한계**: 에이전트가 작업지시자의 로그인된 Dashboard 화면을 직접 확인하지 못할 수 있다. 확인 불가 항목은 "미확정" 또는 "작업지시자 확인 필요"로 표시한다.
- **PR #38 반영 누락**: #9 Store copy 초안과 `PRIVACY.md`에 낡은 large canvas 표현이 남아 있을 수 있다. Stage 1 grep으로 확인하고 필요한 최소 보정을 계획한다.
- **Asset blocker 지속**: Store screenshot과 small promotional image는 이번 task에서 제작하지 않는다. 실제 제출 차단 여부와 요구 규격을 명확히 남긴다.
- **URL 최종성**: privacy policy, homepage, support URL은 PR merge와 게시 방식에 따라 stable URL이 달라질 수 있다. 후보와 최종 승인 필요 항목을 분리한다.
- **검증 비용**: package 생성과 전체 테스트가 시간이 걸릴 수 있다. 제출 문서 변경이라도 final validation에서는 build/typecheck/test를 수행해 package와 권한 경계를 확인한다.
