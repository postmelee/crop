# Task #50 수행계획서 - GitHub Release note 템플릿 표준화

GitHub Issue: [#50](https://github.com/postmelee/crop/issues/50)
마일스톤: M030

## 목적

반복 release에서 재사용할 GitHub Release note 작성 기준을 표준화한다. 사용자에게 보이는 설치/업데이트 안내와 maintainer가 확인해야 하는 release 검증 정보를 분리해, 다음 release부터 `gh release create --notes-file` 입력으로 바로 사용할 수 있는 템플릿과 작성 흐름을 만든다.

이번 task는 Chrome Web Store 게시 완료 이후 반복 배포 운영을 안정화하기 위한 문서/템플릿 작업이다. 기존 `v0.1.0` GitHub Release body 수정, release 자동화 workflow 도입, version bump 정책 변경, Chrome Web Store Dashboard 제출 작업은 수행하지 않는다.

## 배경

첫 release `v0.1.0`은 Chrome Web Store 제출 기준을 만들기 위해 최소 release note로 게시됐다. 작업지시자가 2026년 6월 8일 같은 스레드에서 Chrome Web Store 게시 완료를 확인했으므로, 다음부터는 Store 상태, privacy URL, package asset checksum, 검증 결과를 release note에 일관되게 남겨야 한다.

#46은 `mydocs/manual/release_pipeline_guide.md`에 release PR, tag/GitHub Release, Store 제출 전 검증 흐름을 정리했다. #48은 첫 `main` 기준과 `v0.1.0` GitHub Release asset/ZIP 동기화 근거를 남겼다. #50은 이 흐름을 이어 받아 release note body의 출력 형식과 수동 작성 절차를 고정한다.

## 범위

### 포함

- release note 템플릿 위치 판단과 작성 규칙 문서화
- 사용자 측면 섹션: 설치/업데이트 안내, 주요 변경점, 권한/privacy 변화, 알려진 제한, Chrome Web Store 상태
- 개발자 측면 섹션: release commit/tag, 포함 PR/issue, package 생성 명령, asset checksum, 검증 결과, rollback 또는 후속 작업
- `gh release create --notes-file` 기반 수동 release note 작성 흐름 정리
- `mydocs/manual/release_pipeline_guide.md`와 release note 템플릿 간 상호 링크
- GitHub 공식 자동 release notes category와 `.github/release.yml` 적용 필요성 검토

### 제외

- 기존 `v0.1.0` GitHub Release body 수정
- release 자동화 workflow 도입
- version bump 정책 변경
- Chrome Web Store Dashboard 제출 작업
- Store listing copy, screenshot, promo video, small promotional image 제작 또는 변경
- 앱 기능, 권한, manifest 동작 변경

## 설계 방향

- release note body 템플릿은 특정 release의 실제 로그가 아니라 반복 작성 형식이므로 `mydocs/_templates/` 아래에 둔다.
- release 실행 절차와 `--notes-file` 사용 흐름은 반복 운영 매뉴얼인 `mydocs/manual/release_pipeline_guide.md`에 연결한다.
- GitHub 자동 release notes category는 body 템플릿을 대체하지 않는다. Stage 1에서 공식 문서를 확인한 뒤, PR/issue category 자동화에 실효성이 있을 때만 `.github/release.yml`을 추가한다.
- 사용자 측면 섹션과 개발자 측면 섹션은 release body 안에서 명확히 분리한다.
- release note에는 Chrome Web Store 상태와 privacy URL이 반드시 들어가야 하지만, Dashboard 제출 실행 자체는 이 task 범위 밖으로 둔다.
- checksum과 검증 결과는 release 기준에서 생성한 asset의 재현성과 Store 업로드 후보 동일성을 확인하기 위한 필수 항목으로 둔다.

## 잠정 단계

- **Stage 1 — 공식 기능과 기존 release 흐름 조사**
  - GitHub 공식 자동 release notes 문서와 GitHub CLI `gh release create` manual을 확인한다.
  - 기존 `release_pipeline_guide.md`, #48 보고서, `.github/` 구성을 대조한다.
  - `.github/release.yml`을 이번 task에서 추가할지, release body 템플릿만 둘지 판단한다.
- **Stage 2 — release note 템플릿 작성**
  - 사용자 측면과 개발자 측면을 분리한 `mydocs/_templates/github_release_note.md`를 작성한다.
  - 설치/업데이트, 주요 변경점, 권한/privacy, 알려진 제한, Store 상태, commit/tag, PR/issue, package command, checksum, verification, rollback/follow-up 항목을 포함한다.
  - `_templates/README.md`에 신규 템플릿의 작성 시점과 사용 기준을 연결한다.
- **Stage 3 — release pipeline guide 연결과 선택 설정 반영**
  - `release_pipeline_guide.md`에 release note 작성 순서와 `gh release create --notes-file` 사용 예를 추가한다.
  - Stage 1 판단 결과에 따라 필요 시 `.github/release.yml` category 설정을 추가한다.
  - manual 본문에는 반복 절차만 두고, 특정 release 결과는 단계 보고서/최종 보고서에 남긴다.
- **Stage 4 — 통합 검증과 최종 보고**
  - 템플릿 필수 키워드 grep, 상호 링크 확인, `git diff --check`를 수행한다.
  - 최종 보고서에 확정된 템플릿 위치, `.github/release.yml` 채택 여부, 다음 release 사용 방법을 요약한다.

## 검증 계획

### 단계별 검증

- Stage 1
  - GitHub 공식 release notes 문서 확인
  - GitHub CLI `gh release create` manual 확인
  - `rg -n "Release|release note|gh release|notes-file|Chrome Web Store|checksum|privacy" mydocs/manual mydocs/report .github`
- Stage 2
  - `rg -n "user|developer|checksum|privacy|Chrome Web Store|asset|verification|rollback|known" mydocs/_templates/github_release_note.md mydocs/_templates/README.md`
  - `git diff --check`
- Stage 3
  - `rg -n "notes-file|github_release_note|release.yml|Chrome Web Store|privacy|checksum" mydocs/manual/release_pipeline_guide.md mydocs/_templates .github`
  - 필요 시 `.github/release.yml` 구조 확인
  - `git diff --check`
- Stage 4
  - `rg -n "user|developer|checksum|privacy|Chrome Web Store|asset|verification" mydocs/_templates mydocs/manual .github`
  - `git status --short`
  - `git diff --check`

### 통합 검증

- release note 템플릿 위치와 작성 규칙이 승인된 문서에 남는다.
- 사용자 안내와 개발자 검증 정보가 분리된 형식으로 확정된다.
- asset checksum, privacy URL, Store 상태, 검증 결과 항목이 포함된다.
- 다음 release에서 수동 작성 또는 `--notes-file`로 바로 재사용 가능한 수준이다.
- release pipeline guide 또는 template 문서와 상호 링크가 존재한다.
- 기존 `v0.1.0` GitHub Release body, version bump 정책, Chrome Web Store Dashboard 상태는 변경하지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **템플릿 위치 혼선**: GitHub Release body는 공개 표면이지만 저장소 안에서는 운영 템플릿이다. 수행계획서에서 `mydocs/_templates/` 선택을 승인받고, 실제 공개 release body 수정은 별도 release 작업에서 수행한다.
- **자동 release notes와 수동 body의 역할 혼동**: `.github/release.yml`은 자동 category 설정이고, 사용자/개발자 섹션이 있는 body 템플릿을 대체하지 않는다. Stage 1에서 역할을 분리해 기록한다.
- **manual에 일회성 결과 누적**: `release_pipeline_guide.md`에는 반복 절차만 넣고, 특정 release의 checksum이나 검증 로그는 단계 보고서/최종 보고서에 남긴다.
- **Chrome Web Store 상태 변동**: Store 상태는 release마다 달라질 수 있다. 템플릿에는 상태 입력란과 확인 기준을 두되, 실제 Dashboard 조작은 이번 task에서 제외한다.

## 승인 요청 사항

- release note body 템플릿을 `mydocs/_templates/github_release_note.md`에 두는 방향을 승인한다.
- `gh release create --notes-file` 사용 절차를 `mydocs/manual/release_pipeline_guide.md`에 연결하는 방향을 승인한다.
- `.github/release.yml`은 Stage 1 공식 문서 확인 후 실효성이 있을 때만 추가하고, 불필요하면 보류 판단을 문서화하는 방향을 승인한다.
- 기존 `v0.1.0` GitHub Release body 수정, release 자동화 workflow 도입, version bump 정책 변경, Chrome Web Store Dashboard 작업을 제외하는 범위를 승인한다.
- 위 Stage 1~4 구성과 검증 계획으로 구현계획서 작성을 진행하는 것을 승인한다.

승인되면 `task_m030_50_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
