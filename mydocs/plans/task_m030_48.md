# Task #48 수행계획서 - 첫 main 릴리즈와 Chrome Web Store 제출본 동기화

GitHub Issue: [#48](https://github.com/postmelee/crop/issues/48)
마일스톤: M030

## 목적

첫 Chrome Web Store 제출 전에 `devel`에 병합된 release 준비 상태를 실제 제출 기준으로 승격한다. 현재 원격 저장소에는 `main` 브랜치와 GitHub Release/tag가 없으므로, 제출용 privacy policy URL과 package artifact 기준이 아직 stable하지 않다.

이번 task는 첫 `main` 기준을 만들고, 같은 기준에서 생성한 Chrome Web Store ZIP을 GitHub Release asset과 Dashboard 업로드 후보로 일치시키는 운영 작업이다. 단, Chrome Web Store Dashboard의 실제 `Submit for review` 클릭은 별도 작업지시자 승인 전까지 수행하지 않는다.

## 배경

PR #47은 #46의 release pipeline, PR CI, package 검증, 제출 runbook을 `devel`에 반영했다. #46 문서는 `devel -> main` release PR, `main` 기준 package 재생성, tag/GitHub Release, Dashboard 제출 순서를 정했지만, 현재 저장소는 아직 원격 `main` 브랜치가 없어 표준 release PR 경로가 그대로 성립하지 않는다.

#37 Dashboard 입력값은 privacy policy URL 후보를 `https://github.com/postmelee/crop/blob/main/PRIVACY.md`로 정리했다. 따라서 실제 제출 전에는 `main` 또는 release tag 기준 `PRIVACY.md` URL이 접근 가능해야 하며, 제출 ZIP도 같은 기준에서 다시 만들어야 한다.

## 범위

### 포함

- `devel` 최신 상태, PR #47 merge 결과, 원격 `main`/tag/Release 현황 확인
- 첫 `main` 기준을 만드는 방법 결정과 실행
- release 기준 `npm run build`, `npm run typecheck`, `npm test`, `npm run package:cws`, `npm run verify:cws` 실행
- `/tmp/crop-0.1.0-cws.zip` contents와 manifest 권한 경계 확인
- GitHub tag/Release 생성과 CWS ZIP asset 업로드
- GitHub Release asset과 Dashboard 업로드 후보 파일 동일성 근거 기록
- `main` 또는 tag 기준 `PRIVACY.md` URL 접근 확인
- 작업 단계 보고서, 최종 보고서, 오늘할일 갱신

### 제외

- Chrome Web Store Dashboard의 실제 `Submit for review` 클릭
- Store listing copy, screenshot, promo video, small promotional image 제작 또는 변경
- `debugger`, `<all_urls>`, broad `host_permissions`, `tabs` 권한 추가
- capture 기능, branding, privacy policy 본문 변경
- #46 release pipeline runbook의 구조적 재설계

## 설계 방향

- release 기준은 `devel`에 merge된 PR #47 이후 커밋을 출발점으로 삼는다.
- 원격 `main`이 없기 때문에 Stage 1에서 첫 `main` bootstrap 방식을 먼저 기록하고 승인받는다.
- 가능한 경우 `main` 생성 후 이후 release는 #46 runbook의 `devel -> main` PR 경로로 돌아간다.
- GitHub Release tag는 `manifest.json`의 version `0.1.0`과 맞춰 `v0.1.0` 후보로 둔다.
- Chrome Web Store upload 후보는 release 기준에서 fresh 생성한 `/tmp/crop-0.1.0-cws.zip` 하나로 고정한다.
- GitHub Release asset과 Dashboard upload 후보의 동일성은 파일 크기와 checksum으로 확인한다.
- Dashboard 입력은 작업지시자가 직접 수행할 수 있도록 URL, asset, checksum, 중단 조건을 최종 보고서에 남긴다.

## 잠정 단계

- **Stage 1 — 첫 main 기준과 release 경로 확정**
  - 원격 브랜치, default branch, PR #47 merge, tag/Release 현황을 확인한다.
  - `main`이 없는 첫 release bootstrap 방법과 승인 지점을 기록한다.
  - 검증 관점은 release 기준 커밋이 PR #47 merge 결과와 일치하는지 확인하는 것이다.
- **Stage 2 — release 기준 package 재생성과 검증**
  - release 기준 checkout에서 build/typecheck/test/package/verify를 실행한다.
  - ZIP contents, manifest 권한, forbidden entry absence, checksum을 기록한다.
  - 검증 관점은 Chrome Web Store에 업로드 가능한 단일 ZIP 후보가 만들어졌는지 확인하는 것이다.
- **Stage 3 — GitHub Release asset과 privacy URL 동기화**
  - 승인된 기준으로 tag/GitHub Release를 만들고 CWS ZIP asset을 업로드한다.
  - `main` 또는 tag 기준 `PRIVACY.md` URL 접근성과 Release asset checksum을 확인한다.
  - 검증 관점은 Dashboard 입력값과 업로드 파일이 release 기준과 일치하는지 확인하는 것이다.
- **Stage 4 — 최종 제출 전 보고와 승인 대기**
  - 최종 보고서에 privacy URL, Release URL, asset checksum, Dashboard upload 후보, submit 중단 조건을 정리한다.
  - Chrome Web Store `Submit for review`는 누르지 않고 작업지시자에게 별도 제출 승인을 요청한다.

## 검증 계획

### 단계별 검증

- Stage 1
  - `git status --short --branch`
  - `git branch -vv --all`
  - `git ls-remote --heads origin main devel`
  - `gh pr view 47 --repo postmelee/crop --json number,state,baseRefName,headRefName,mergeCommit,url`
  - `gh release list --repo postmelee/crop --limit 10`
- Stage 2
  - `npm run build`
  - `npm run typecheck`
  - `npm test`
  - `npm run package:cws`
  - `npm run verify:cws`
  - `unzip -l /tmp/crop-0.1.0-cws.zip`
  - `unzip -Z1 /tmp/crop-0.1.0-cws.zip`
  - checksum 확인
- Stage 3
  - `git tag --list`
  - GitHub Release/tag 확인
  - GitHub Release asset 확인
  - `main` 또는 tag 기준 `PRIVACY.md` URL 확인
  - asset checksum과 local ZIP checksum 대조
- Stage 4
  - `rg -n "Submit for review|PRIVACY.md|crop-0.1.0-cws.zip|v0.1.0|checksum" mydocs/working mydocs/report`
  - `git diff --check`

### 통합 검증

- `main` 또는 release tag 기준 `PRIVACY.md` URL이 접근 가능한 제출 후보로 확인된다.
- release 기준 `/tmp/crop-0.1.0-cws.zip`가 `verify:cws`를 통과한다.
- ZIP root에 `manifest.json`이 있고 repository root 문서/config, `mydocs`, `node_modules`, `.DS_Store`, `__MACOSX`가 없다.
- manifest 권한이 `activeTab`, `scripting`, `clipboardWrite`, `downloads`를 유지한다.
- GitHub Release asset과 Dashboard upload 후보 파일이 동일하다는 checksum 근거가 있다.
- Chrome Web Store `Submit for review`를 수행하지 않았음을 최종 보고서에 명시한다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **첫 `main` 부트스트랩 예외**: #46 runbook은 이미 `main`이 있는 상황을 전제로 한다. 현재는 `main`이 없으므로 첫 생성 방식은 Stage 1에서 별도 승인받아야 한다.
- **Dashboard 실제 UI 불일치**: Chrome Web Store Dashboard의 필수 asset, deferred publishing, privacy field UI가 문서와 다를 수 있다. 이 task는 제출 버튼 클릭을 제외하고, 최종 보고서에서 작업지시자 직접 확인 항목으로 남긴다.
- **Release asset과 upload 후보 불일치**: ZIP을 여러 번 만들면 파일이 달라질 수 있다. release 기준 fresh ZIP 하나를 만들고 checksum으로 동일성을 고정한다.
- **network/API 작업 실패**: GitHub Release 생성과 asset 업로드는 네트워크와 인증에 의존한다. 실패 시 실패 지점을 보고하고 재시도 또는 수동 대체 절차를 승인받는다.

## 승인 요청 사항

- 위 범위와 단계 구성으로 #48을 진행한다.
- 첫 `main`이 없는 상황을 Stage 1에서 별도 release bootstrap 판단으로 다룬다.
- release tag 후보를 `v0.1.0`으로 두고, 최종 tag 생성 전 Stage 3에서 다시 확인한다.
- Chrome Web Store `Submit for review`는 이번 task 완료 후에도 별도 작업지시자 승인 전 수행하지 않는다.

승인되면 `task_m030_48_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
