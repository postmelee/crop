# Task #29 수행계획서 - GitHub Community Standards 보강

GitHub Issue: [#29](https://github.com/postmelee/crop/issues/29)
마일스톤: M030

## 목적

`crop` 공개 저장소가 GitHub Community Standards에서 요구하는 기본 기여 안내와 안전 정책을 갖추도록 정리한다. `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, issue template 인식 문제를 중심으로 현재 저장소 성격에 맞는 community health file을 보강한다.

또한 하이퍼-워터폴 내부 절차와 외부 기여자 안내가 서로 충돌하지 않도록 문서 경계를 명확히 한다. 외부 기여자는 공개 저장소의 최소 절차를 이해할 수 있어야 하고, 작업 에이전트와 내부 작업자는 기존 `mydocs/manual/`과 task 산출물을 계속 진실 원천으로 사용할 수 있어야 한다.

추가로 GitHub Sponsors profile이 존재하는 maintainer 계정(`postmelee`)을 repository sponsor button으로 연결할 수 있도록 `.github/FUNDING.yml`을 추가한다. GitHub Discussions는 이미 저장소에서 활성화되어 있으므로, 실제 discussion 게시나 category 설정 변경은 수행하지 않고 `crop` 소개와 커뮤니티 대화 시작에 사용할 게시글 후보만 작업 산출물로 준비한다.

## 배경

M030 #9 배포 준비 과정에서 공개 저장소의 Community Standards checklist 보강 필요가 확인되었다. #29 이슈 본문 기준으로 GitHub Community API health percentage는 57이며, `README`, `LICENSE`, `pull_request_template`은 인식되지만 `code_of_conduct`, `contributing`, `documentation`, `issue_template`이 비어 있었다.

2026-06-08 현재 `gh api repos/postmelee/crop/community/profile` 재확인 결과 health percentage는 여전히 57이다. Chrome Web Store 게시 완료 이후 `documentation`은 `https://chromewebstore.google.com/detail/crop/pdmniipgbjdcpnhbkkppodechbehagki`로 인식되고 있으나, `code_of_conduct`, `contributing`, `issue_template`은 아직 `null`이다.

저장소에는 `.github/ISSUE_TEMPLATE/task.yml`이 이미 존재한다. 따라서 이번 task는 단순히 문서를 추가하는 데 그치지 않고, issue template이 community profile에서 인식되지 않는 원인을 `.github` 구조와 GitHub form schema 관점에서 점검해야 한다.

## 설계 방향

- GitHub가 자동 인식하는 표준 위치를 우선 사용한다. 공개 기여자용 문서는 루트 `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, 필요 시 `SECURITY.md`, `SUPPORT.md`에 둔다.
- `.github/ISSUE_TEMPLATE/task.yml`은 기존 하이퍼-워터폴 task 등록 목적을 유지하되, GitHub issue forms 인식 조건에 맞지 않는 부분이 있으면 최소 수정한다.
- 외부 기여자 안내는 "모든 기여자가 내부 에이전트 절차를 수행해야 한다"는 식으로 쓰지 않는다. 대신 issue 기반 논의, PR template, maintainer 확인, scope 분리 원칙을 공개 저장소 언어로 설명한다.
- `AGENTS.md`와 `mydocs/manual/`은 작업 에이전트와 내부 작업자를 위한 운영 규칙으로 유지한다. 공개 문서는 이 절차를 요약하거나 링크하되, 내부 산출물 작성 규칙을 과도하게 노출하지 않는다.
- `CODE_OF_CONDUCT.md`는 공개 저장소에 필요한 최소 행동 기준과 연락/신고 경로를 포함하되, 법률 검토가 필요한 조직 정책처럼 확장하지 않는다.
- `SECURITY.md`와 `SUPPORT.md`는 GitHub Community Standards와 공개 확장 운영에 실익이 있는지 Stage 1에서 판단한다. 추가한다면 취약점/지원 요청 경로와 비범위를 짧게 고정한다.
- `.github/FUNDING.yml`은 GitHub 공식 sponsor button 인식 위치인 `.github/`에 두고, GitHub Sponsors 계정은 `postmelee`로 고정한다. 다른 funding platform이나 custom URL은 이번 task에서 추가하지 않는다.
- Discussions 게시글 후보는 실제 GitHub Discussions에 게시하지 않고 `mydocs/working/` 작업 산출물로 작성한다. 게시 후보는 `crop` 소개, 설치/사용 안내, 피드백 요청, 로드맵/제약 공유, sponsor 안내처럼 public discussion으로 옮길 수 있는 짧은 글 묶음으로 둔다.
- README 링크 정리는 community health file discoverability를 돕는 수준으로 제한한다. #51의 Chrome Web Store 게시 완료 README 갱신과 충돌하면 #51 병합 결과를 우선 반영하고 #29에서 rebase 후 정리한다.

## 잠정 단계

- **Stage 1 — Community Profile 감사와 문서 범위 확정**
  - `gh api repos/postmelee/crop/community/profile`, GitHub Community Standards page, `.github/ISSUE_TEMPLATE/task.yml`, `.github/pull_request_template.md`, README를 확인한다.
  - `code_of_conduct`, `contributing`, `issue_template`, `security`, `support` 각각을 추가/수정/보류할지 확정한다.
  - 검증 관점은 실제 누락 항목과 이번 task에서 다룰 파일 범위가 일치하는지 확인하는 것이다.

- **Stage 2 — 기여자 안내와 행동 강령 작성**
  - `CONTRIBUTING.md`와 `CODE_OF_CONDUCT.md`를 English canonical로 작성하고, 한국어/중국어/일본어 번역 파일을 작성한다.
  - 기여 흐름, issue/PR 기대사항, Hyper-Waterfall과 외부 기여 경계, review/merge 원칙, 신고/연락 경로를 정리한다.
  - 검증 관점은 공개 기여자가 이해할 수 있으면서 내부 `AGENTS.md`/`mydocs/manual/`과 충돌하지 않는지 확인하는 것이다.

- **Stage 3 — Issue template 인식 문제 수정**
  - `.github/ISSUE_TEMPLATE/task.yml`의 GitHub issue forms schema와 community profile 인식 조건을 점검한다.
  - 필요한 경우 `config.yml` 또는 template metadata를 최소 수정한다.
  - 검증 관점은 GitHub API 또는 수동 Community Standards page에서 issue template 인식 상태가 개선되는지 확인하는 것이다.

- **Stage 4 — 보안/지원/README 링크 정리와 최종 검증**
  - Stage 1 판단에 따라 `SECURITY.md`, `SUPPORT.md`를 추가하거나 보류 사유를 남긴다.
  - `.github/FUNDING.yml`을 추가하고 GitHub Sponsors 계정 `postmelee`를 연결한다.
  - GitHub Discussions에 게시할 `crop` 소개와 커뮤니티 게시글 후보를 `mydocs/working/`에 정리한다.
  - README와 community 문서 간 링크를 최소 범위로 정리한다.
  - 검증 관점은 Community Standards checklist, 문서 링크, 하이퍼-워터폴 문서 경계가 일관되는지 확인하는 것이다.

## 리스크

- **GitHub 인식 지연**: community profile API와 Community Standards page가 파일 추가 직후 바로 갱신되지 않을 수 있다. API 결과, GitHub UI 수동 확인 시점, 남은 지연 가능성을 단계/최종 보고서에 구분해 기록한다.
- **외부 기여 안내 과잉**: 내부 작업자용 하이퍼-워터폴 절차를 그대로 공개 문서에 요구하면 외부 기여 진입 장벽이 불필요하게 높아질 수 있다. 공개 문서는 issue/PR 기반 기대사항으로 요약하고 내부 산출물 규칙은 링크/참조 수준으로 제한한다.
- **정책 문서의 법률적 과확정**: `CODE_OF_CONDUCT.md`, `SECURITY.md`가 법률 자문이 필요한 약속처럼 읽힐 수 있다. 최소 운영 기준과 연락 경로 중심으로 작성하고 법률/라이선스 변경은 범위 밖에 둔다.
- **#51 README 변경과 충돌**: #51이 Chrome Web Store 게시 완료 후 README 설치 안내를 갱신 중이므로 #29의 README 링크 정리와 같은 날짜 orders 파일에서 충돌할 수 있다. #29는 별도 worktree에서 진행하고, #51 병합 결과를 rebase 후 반영한다.
- **Issue template 원인 불확실성**: `.github/ISSUE_TEMPLATE/task.yml`이 default branch 상태, branch visibility, form schema, GitHub 캐시 중 어느 원인으로 인식되지 않는지 즉시 단정하기 어려울 수 있다. Stage 1에서 원인 후보를 분리하고 확인 가능한 수정부터 적용한다.
- **Discussion 게시와 문안의 분리**: 실제 GitHub Discussions 게시글 생성은 원격 플랫폼 쓰기 작업이고 기존 제외 범위인 Discussions 설정 변경과도 경계가 가깝다. 이번 task에서는 후보 문안만 만들고 실제 게시 여부는 PR 이후 별도 승인으로 분리한다.
- **Funding 계정 오기입**: `.github/FUNDING.yml`의 `github` 값이 Sponsors profile 계정과 다르면 sponsor button이 잘못 연결될 수 있다. Stage 4에서 인증 사용자 `postmelee`와 파일 값을 대조한다.
