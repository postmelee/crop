# Task #26 수행계획서

GitHub Issue: [#26](https://github.com/postmelee/crop/issues/26)
마일스톤: M020

## 목적

선택 영역을 만든 뒤 페이지를 스크롤해도 Save/Copy 결과가 선택한 page rect 전체 크기를 유지하도록 보정한다. Firefox 원본 기능처럼 현재 스크롤 상태의 sticky/header 픽셀이 섞일 수는 있어도, 출력 이미지의 영역과 크기는 사용자가 확정한 선택 박스와 일치해야 한다.

이번 task는 기존 visible viewport 기반 선택 캡처의 빠른 경로를 유지하면서, 선택 rect가 현재 viewport를 벗어난 경우에만 선택 영역 전용 tile capture/stitching 경로를 추가하는 방향으로 진행한다.

## 배경

Task #24 검증 중 phase6 edge fixture에서 선택 영역을 만든 뒤 스크롤하고 저장하면 저장 결과가 선택 박스 전체가 아니라 현재 visible viewport와의 교집합만 저장되는 문제가 확인됐다. 비교 결과 Firefox 원본은 `1,520 x 920` CSS px 선택을 DPR 2 기준 `3,040 x 1,840` px PNG로 저장했지만, crop은 `2,162 x 1,520` px로 잘린 결과를 만들었다.

현재 `src/content/overlay/crop-overlay.ts`의 selected capture 경로는 `clipPageRectToViewport(selectedRect, viewport)`를 먼저 적용한 뒤 `chrome.tabs.captureVisibleTab()` 결과를 crop한다. 따라서 선택 rect가 스크롤로 viewport 밖으로 일부 이동하면 offscreen 부분은 캡처 원본에 없고, 최종 PNG도 줄어든다.

Task #15에서 full page tile capture/stitching 경로가 이미 도입되어 있으므로, 이 task는 그 로직의 좌표/타일/stitch 계약을 선택 rect 범위에 재사용할 수 있는지 검토하고 필요한 최소 확장을 수행한다.

## 범위

### 포함

- selected Save/Copy 경로의 viewport clipping 동작 수정
- 선택 rect가 현재 viewport를 벗어난 경우 선택 rect 범위만 tile capture/stitching하는 경로 추가
- 기존 full page capture/stitching helper의 재사용 가능 범위 정리 및 필요한 최소 추상화
- overlay, selection outline, handles, action buttons가 최종 PNG에 포함되지 않도록 capture visibility 처리 유지
- capture 성공/실패와 무관한 scroll restoration 검증
- phase6 fixture 또는 자동 테스트로 스크롤 후 선택 영역 저장 회귀 보강
- 품질 매트릭스 또는 관련 내부 작업 문서 갱신

### 제외

- `debugger` 권한 추가
- `<all_urls>` 또는 broad host permission 추가
- full page capture 기능 자체의 UX/정책 확장
- Task #24의 큰 wrapper 자동 선택 후보 정책 변경
- cross-origin iframe 내부 문서의 별도 scroll stitching
- Firefox privileged `drawSnapshot`/브라우저 내부 API 포팅
- fixed/sticky element를 Firefox와 픽셀 단위로 완전 동일하게 만드는 보정

## 설계 방향

- 현재 viewport 안에 완전히 들어온 선택 영역은 기존 `captureVisibleTab()` + 단일 crop 경로를 유지한다. 불필요한 스크롤 이동과 stitching 비용을 피하기 위해서다.
- 선택 rect가 현재 viewport를 벗어나면 선택 rect를 page 좌표 기준 bounds로 삼고, viewport 크기에 맞춰 필요한 tile만 만든다.
- tile capture는 기존 full page capture처럼 `scrollTo()`, paint 대기, overlay 숨김, `captureVisibleTab()`, 원래 scroll 위치 복구 순서를 따른다.
- stitching destination은 선택 rect의 좌상단을 `(0, 0)`으로 둔다. 최종 CSS 크기는 선택 rect 크기이고, 실제 PNG 픽셀 크기는 캡처 이미지와 viewport CSS size 비율로 결정한다.
- 기존 full page helper를 무리하게 일반화해 full page 동작을 흔들지 않는다. 공통화가 필요하면 `bounds`를 받는 tile plan helper처럼 좁은 단위로 추출한다.
- sticky/fixed 요소는 scroll stitching 방식의 한계를 인정한다. Firefox 원본처럼 현재 스크롤 상태의 header가 결과에 섞일 수 있으나, 이번 task의 핵심 수용 기준은 선택 rect 전체 크기와 offscreen clipping 방지다.
- capture 중 화면이 순간 이동하는 현상은 선택 rect가 viewport 밖으로 나간 경우에는 기술적으로 발생할 수 있다. 다만 선택 rect 범위로 tile 수를 최소화하고, 완료/실패 후 시작 scroll 위치로 복구한다.

  - 자동 테스트와 수동 smoke 기준을 추가하고 품질 매트릭스에 #26 검증 항목을 반영한다.

- **Stage 4 — 통합 검증과 최종 보고**
  - build/typecheck/test, 권한 grep, diff check를 실행한다.
  - 결과 크기와 overlay 미포함 확인, 남은 scroll-stitching 한계를 최종 보고서에 기록한다.

## 리스크

- **화면 이동 노출**: 선택 rect가 viewport 밖으로 나간 경우 tile capture를 위해 scroll 이동이 필요하다. 선택 rect 범위만 캡처해 이동 횟수를 줄이고, 완료/실패 후 scroll restoration을 보장한다.
- **full page 경로 회귀**: 기존 full page helper를 재사용하다가 전체 페이지 캡처 계약을 흔들 수 있다. 공통화는 bounds/tile plan 같은 작은 단위로 제한하고 기존 full page 테스트를 함께 실행한다.
- **sticky/fixed 픽셀 차이**: Chrome MV3의 visible tab capture 기반 stitching은 Firefox privileged snapshot과 동일하지 않다. 이번 task는 출력 rect 크기와 clipping 방지를 우선 수용 기준으로 둔다.
- **layout shift**: capture 중 lazy loading, scroll anchoring, sticky header가 layout을 바꿀 수 있다. fixture와 안정 문서를 기준으로 검증하고 실제 웹 잔여 위험은 보고서에 남긴다.
- **큰 선택 영역**: 선택 rect가 매우 크면 canvas 한계 또는 capture 시간이 문제가 될 수 있다. 기존 full page capture의 출력 크기 제한 정책을 재사용한다.
