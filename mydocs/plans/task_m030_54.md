# Task #54 수행계획서 - v0.1.0 GitHub Release body 소급 표준화

GitHub Issue: [#54](https://github.com/postmelee/crop/issues/54)
마일스톤: M030

## 목적

#50에서 확정한 GitHub Release note 템플릿과 `gh release create --notes-file` 작성 흐름을 기존 `v0.1.0` GitHub Release body에 소급 적용한다. 사용자 안내와 developer 검증 기록을 분리하고, Chrome Web Store 상태, privacy URL, release asset, SHA-256 checksum, verification 결과를 기존 release 기준으로 명확히 남긴다.

이번 task는 기존 GitHub Release body를 보정하는 운영 작업이다. 새 tag, 새 GitHub Release, release asset 교체, version bump, Chrome Web Store Dashboard 작업은 수행하지 않는다.

## 배경

#50에서는 release note 템플릿과 release pipeline guide를 표준화했지만, 수행계획서의 제외 범위에 따라 기존 `v0.1.0` GitHub Release body는 수정하지 않았다. 작업지시자가 기존 release도 새 규칙을 따르는 것이 맞다고 판단했고, 해당 작업은 #54로 분리됐다.

#48 최종 보고서는 `v0.1.0` release 기준 commit, Release URL, asset URL, ZIP SHA-256, privacy URL 후보를 기록했다. #54는 그 근거와 실제 GitHub Release 현재 상태를 다시 확인한 뒤, #50 템플릿 구조에 맞는 body 초안을 만들고 승인 후 원격 GitHub Release body만 업데이트한다.

## 범위

### 포함

- `v0.1.0` GitHub Release body 현황 확인
- `v0.1.0` tag, release commit, asset URL, asset size, SHA-256 checksum, privacy URL 확인
- #50 템플릿 기준 `v0.1.0` release body 초안 작성
- 작업지시자 승인 후 `v0.1.0` GitHub Release body 업데이트
- 업데이트 전후 body 요약, 검증 결과, 남은 리스크 문서화

### 제외

- 새 tag 또는 새 GitHub Release 생성
- release asset 교체, 삭제, 재업로드
- version bump 정책 변경
- release 자동화 workflow 도입
- Chrome Web Store Dashboard 제출, upload, 상태 변경
- #50 템플릿 구조 재설계
- README, manifest, privacy policy 본문 변경

## 설계 방향

- `v0.1.0` Release body는 원격 GitHub 플랫폼 산출물이므로 Stage 3에서 별도 승인 후 수정한다.
- Stage 1에서는 읽기 작업만 수행해 현재 release body와 asset 상태를 기록한다.
- Stage 2에서는 #50 템플릿 구조를 기준으로 초안을 만들고, 원격 update 전 작업지시자 승인을 받는다.
- Stage 3에서는 승인된 초안만 GitHub Release body에 적용한다. asset, tag, release title, draft/prerelease 상태는 변경하지 않는다.
- Stage 4에서는 원격 release body가 필수 키워드와 기준값을 포함하는지 검증하고 최종 보고한다.

## 잠정 단계

- **Stage 1 — 현재 release 상태와 기준값 확인**
  - `gh release view`, release asset metadata, privacy URL, #48/#50 문서를 대조한다.
  - 현재 body, tag, asset, checksum, Store published 상태 후보를 기술 노트에 기록한다.
  - 원격 Release 수정은 수행하지 않는다.
- **Stage 2 — v0.1.0 release body 초안 작성**
  - `mydocs/_templates/github_release_note.md` 구조로 `v0.1.0` body 초안을 작성한다.
  - 공개 언어, 체크박스 삭제 여부, 포함 PR/issue 표현을 확정 후보로 정리한다.
  - Stage 2 완료 후 원격 release body update 승인을 요청한다.
- **Stage 3 — 승인된 body를 GitHub Release에 반영**
  - 승인된 초안만 `v0.1.0` GitHub Release body에 적용한다.
  - tag, title, asset, draft/prerelease 상태가 바뀌지 않았는지 확인한다.
  - release body 필수 키워드와 checksum/privacy/asset/verification 값을 검증한다.
- **Stage 4 — 통합 검증과 최종 보고**
  - update 전후 요약, 수용 기준, 제외 범위 준수 여부를 최종 보고서에 정리한다.
  - release asset, tag, version, Dashboard 상태가 변경되지 않았음을 확인한다.

## 검증 계획

### 단계별 검증

- Stage 1
  - `gh release view v0.1.0 --repo postmelee/crop --json tagName,name,url,isDraft,isPrerelease,assets,body`
  - `gh release download v0.1.0 --repo postmelee/crop --pattern 'crop-0.1.0-cws.zip' --dir {검증용 임시 경로}`
  - `shasum -a 256 {검증용 임시 경로}/crop-0.1.0-cws.zip`
  - `gh api repos/postmelee/crop/contents/PRIVACY.md?ref=v0.1.0 --jq '.html_url, .sha'`
  - `git diff --check`
- Stage 2
  - `rg -n "user|developer|checksum|privacy|Chrome Web Store|asset|verification|SHA-256" mydocs/tech/task_m030_54_v010_release_body.md`
  - `git diff --check`
- Stage 3
  - `gh release view v0.1.0 --repo postmelee/crop --json tagName,name,url,isDraft,isPrerelease,assets,body`
  - release body grep: `user`, `developer`, `checksum`, `privacy`, `Chrome Web Store`, `asset`, `verification`, `SHA-256`
  - asset URL/size/checksum 재확인
  - `git diff --check`
- Stage 4
  - `rg -n "v0.1.0|checksum|privacy|Chrome Web Store|asset|verification|변경하지|#54" mydocs/tech mydocs/working mydocs/report`
  - `git status --short`
  - `git diff --check`

### 통합 검증

- `v0.1.0` GitHub Release body가 #50 템플릿의 사용자 안내와 developer 검증 기록 분리 구조를 따른다.
- Chrome Web Store 상태, privacy URL, release asset, SHA-256 checksum, verification 결과가 body에 포함된다.
- release asset과 tag/commit 기준은 기존 `v0.1.0` release와 일치한다.
- release body 수정 전후 차이와 검증 결과가 task 산출물에 남는다.
- release asset, tag, version, Dashboard 상태는 변경되지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **원격 release body 수정 리스크**: GitHub Release body는 공개 산출물이다. Stage 2에서 초안을 승인받고 Stage 3에서만 수정한다.
- **기존 release 기준값 불일치**: #48 보고서와 현재 GitHub Release metadata가 다를 수 있다. Stage 1에서 현재 원격 상태를 진실 원천으로 다시 확인한다.
- **asset 변경 오해**: 이번 task는 body만 수정한다. asset URL, size, checksum을 전후 대조해 asset이 바뀌지 않았음을 기록한다.
- **체크박스/내부 문구 공개 위험**: #50 템플릿의 작성 체크는 공개 body에 남길지 삭제할지 Stage 2에서 명시하고 승인받는다.

## 승인 요청 사항

- 위 범위와 Stage 1~4 구성으로 #54를 진행하는 것을 승인한다.
- 원격 GitHub Release body 수정은 Stage 2 초안 승인 후 Stage 3에서만 수행하는 절차를 승인한다.
- `mydocs/tech/task_m030_54_v010_release_body.md`에 현재 body, 승인 초안, update 전후 비교를 보관하는 위치 판단을 승인한다.
- 새 tag, 새 Release, release asset 변경, version bump, Chrome Web Store Dashboard 작업을 제외하는 범위를 승인한다.

승인되면 `task_m030_54_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
