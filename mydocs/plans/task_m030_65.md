# 수행계획서

GitHub Issue: [#65](https://github.com/postmelee/crop/issues/65)
마일스톤: M030

## 목적

사용자가 crop 사용 중 발견한 버그를 GitHub Issue로 쉽게 제보할 수 있도록 사용자 버그 제보용 Issue Form을 추가한다.

이 작업은 내부 하이퍼-워터폴 작업 생성용 `task.yml`과 별도로, 실제 사용자와 외부 기여자가 버그 내용, 재현 절차, 환경 및 버전, 스크린샷을 구조적으로 남길 수 있는 입력 표면을 제공하는 것이 목적이다.

## 배경

사용자가 macOS와 Chrome 환경에서 복사와 다운로드 이미지 오른쪽에 불필요한 흰색 마진이 생긴다고 제보했다. 현재 저장소에는 내부 작업 추적을 위한 `.github/ISSUE_TEMPLATE/task.yml`만 있어, 일반 사용자가 버그를 제보할 때 필요한 재현 정보와 환경 정보를 누락하기 쉽다.

GitHub Issue Form은 GitHub 플랫폼 템플릿이므로 `.github/ISSUE_TEMPLATE/`에서 관리한다. 실제 흰색 마진 버그 수정은 이번 task의 범위가 아니며, 이번 task는 향후 제보 품질을 높이는 템플릿 추가에 집중한다.

## 범위

### 포함

- `.github/ISSUE_TEMPLATE/bug_report.yml` 신규 추가
- 버그 내용, 재현 절차, 기대 결과, 실제 결과, 사용 환경 및 버전, 스크린샷/녹화 자료 입력 항목 구성
- 사용자 제보 예시를 반영한 placeholder 작성
- 기존 내부 작업용 `task.yml`과 역할이 구분되는 이름과 설명 작성

### 제외

- 복사/다운로드 이미지 오른쪽 흰색 마진 버그 수정
- 캡처 백엔드, crop 처리, 오버레이 UI 구현 변경
- 새 GitHub label 또는 milestone 생성
- 제품 공식 문서 루트 신설

## 설계 방향

- 기존 `.github/ISSUE_TEMPLATE/task.yml`과 같은 GitHub Issue Form YAML 형식을 따른다.
- 템플릿 이름과 설명은 사용자가 버그 제보용임을 바로 이해할 수 있게 작성한다.
- 필수 항목은 제보 재현과 triage에 필요한 최소 정보 위주로 지정하고, 스크린샷/녹화 자료는 선택 항목으로 둔다.
- 환경 정보는 OS, 브라우저, crop 버전, 설치 경로 또는 빌드 출처를 분리해 받는다.
- placeholder에는 이번 제보처럼 `macOS`, `Chrome`, 복사/다운로드 결과의 오른쪽 흰색 마진 사례를 예시로 포함한다.

## 잠정 단계

- **Stage 1 — Issue Form 설계 확정**
  - 산출물: 구현계획서에 입력 항목, 필수 여부, placeholder 문구 확정
  - 검증 관점: 내부 작업용 `task.yml`과 사용자 버그 제보용 `bug_report.yml` 역할이 명확히 분리되는지 확인
- **Stage 2 — 버그 제보 템플릿 추가**
  - 산출물: `.github/ISSUE_TEMPLATE/bug_report.yml`
  - 검증 관점: GitHub Issue Form YAML 구조와 placeholder가 수용 기준을 충족하는지 확인
- **Stage 3 — 검증 및 최종 보고**
  - 산출물: 단계 보고서, 최종 보고서, PR 준비 변경 묶음
  - 검증 관점: `git diff --check`, YAML 파싱, 최종 diff 검토

## 검증 계획

### 단계별 검증

- Stage 1
  - `.github/ISSUE_TEMPLATE/task.yml`과 신규 `bug_report.yml` 설계가 중복되지 않는지 검토
- Stage 2
  - `ruby -e 'require "yaml"; YAML.load_file(".github/ISSUE_TEMPLATE/bug_report.yml")'`
  - Issue Form 필수 입력과 placeholder 수동 검토
- Stage 3
  - `git diff --check`
  - `git status --short`

### 통합 검증

- GitHub Issue 생성 화면에서 사용자 버그 제보 템플릿으로 이해 가능한 이름과 설명을 제공한다.
- 버그 내용, 재현 절차, 기대 결과, 실제 결과, 사용 환경 및 버전, 스크린샷/녹화 자료 입력 항목이 포함된다.
- 내부 Hyper-Waterfall task 템플릿과 혼동되지 않는다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **Issue Form 문법 오류**: YAML 파싱 검증과 기존 `task.yml` 형식 비교로 줄인다.
- **사용자 입력 부담 증가**: 필수 항목은 재현과 triage에 필요한 최소한으로 제한하고, 보조 자료는 선택 항목으로 둔다.
- **내부 task 템플릿과 혼동**: 이름, 설명, 제목 prefix를 사용자 버그 제보 목적에 맞춰 분리한다.

## 승인 요청 사항

- `.github/ISSUE_TEMPLATE/bug_report.yml`을 신규 GitHub 플랫폼 템플릿으로 추가하는 방향 승인
- 실제 흰색 마진 버그 수정은 이번 task에서 제외하는 범위 승인
- 위 3단계 잠정 단계와 검증 계획 승인

승인되면 `task_m030_65_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
