# Task #15 수행계획서

GitHub Issue: [#15](https://github.com/postmelee/crop/issues/15)
마일스톤: M020

## 목적

`crop`의 `전체 페이지 선택` placeholder를 실제 full page capture 동작으로 연결한다. 사용자가 전체 페이지 capture를 실행하면 현재 문서의 전체 세로 범위를 여러 viewport screenshot으로 캡처하고, 이를 하나의 PNG로 stitching해 Copy/Save 경로에 전달할 수 있어야 한다.

이번 task는 기존 visible viewport capture/crop 경로를 대체하지 않는다. 기존 요소 선택, drag selection, Copy/Save, same-origin iframe selection 동작은 유지하면서 full page mode를 별도 capture backend로 추가한다.

## 배경

현재 `crop`은 Chrome MV3 확장으로 `chrome.tabs.captureVisibleTab()` 기반 visible viewport capture만 지원한다. README와 AGENTS 기준에서 full page capture와 scroll stitching은 MVP 범위 밖 후속 작업으로 남아 있었고, mode toolbar의 `전체 페이지 선택` 버튼도 비활성 placeholder다.

#15는 Firefox Screenshots에 가까운 기능을 위해 full page capture backend, scroll orchestration, stitching 품질 검증을 분리해 진행하는 작업이다. 단, Firefox privileged API와 Chrome `debugger`, `<all_urls>` 권한 추가는 이번 계획의 기본 방향에 포함하지 않는다. 권한 확대가 필요하다고 판단되면 구현을 중단하고 별도 승인으로 분리한다.

## 범위

### 포함

- full page capture strategy와 scroll stitching 데이터 모델 설계
- 현재 top-level page의 전체 문서 높이를 tile 단위 viewport capture로 저장하는 capture loop
- `전체 페이지 선택` 버튼 활성화와 Copy/Save 경로 연결
- overlay/toolbar/action UI가 최종 PNG에 포함되지 않도록 capture 전/후 visibility 처리
- devicePixelRatio, browser zoom, 마지막 viewport partial tile, horizontal scroll 기본 정책 테스트
- sticky/fixed element 중복 노출 정책 정리와 최소 보정
- Phase 6 fixture 또는 신규 full-page fixture 기반 자동/수동 smoke

### 제외

- 기존 visible viewport element/drag capture 경로의 대체
- `debugger` 권한, `<all_urls>` host permission, broad host permission 추가
- cross-origin iframe 내부 full-page stitching
- iframe 내부 document full-page capture
- closed shadow DOM 내부 접근
- 외부 서버 업로드, telemetry
- PDF/비HTML 특수 페이지 지원
- Firefox privileged `drawSnapshot` 직접 포팅

## 설계 방향

- 권한은 현행 `activeTab`, `scripting`, `clipboardWrite`, `downloads` 범위에서 시작한다.
- content script가 page scroll 위치와 document dimensions를 읽고, background service worker는 기존처럼 `chrome.tabs.captureVisibleTab()` 호출만 담당한다.
- full page capture는 page 좌표 기준 tile plan을 만든 뒤, 각 tile마다 window scroll을 이동하고 다음 paint 이후 viewport PNG를 받아 canvas에 stitch한다.
- image scale은 실제 captured image natural size와 viewport CSS size의 비율로 계산한다. devicePixelRatio를 직접 가정하지 않는다.
- 마지막 tile은 문서 bottom에 맞게 viewport crop 영역을 줄여 stitching canvas에 배치한다.
- overlay root는 전체 capture loop 동안 숨기고, 실패/취소 시 원래 상태로 복구한다.
- scroll restoration은 필수다. capture 성공/실패와 무관하게 사용자가 시작했던 scroll position으로 되돌린다.
- sticky/fixed element는 Firefox와 완전히 같은 privileged capture가 불가능하므로, Stage 1에서 MVP 정책을 확정한다. 기본 후보는 "first tile에는 정상 포함, 이후 tile에서는 중복이 눈에 띄지 않도록 fixed/sticky overlay 보정 또는 제한 문서화"다.
- page가 capture 중 layout shift/scroll anchoring을 일으키는 경우를 잔여 위험으로 기록하고, fixture smoke에서 안정적인 문서를 우선 검증한다.

## Firefox 코드 조사 반영

Firefox Screenshots 코드는 #15에서 직접 포팅하지 않고 full page capture/stitching의 설계 reference로 사용한다. 조사 기준 파일은 다음과 같다.

- `browser/components/screenshots/ScreenshotsUtils.sys.mjs`
  - `takeScreenshot(browser, "FullPage")`가 full page bounds를 얻은 뒤 canvas 생성 경로로 진입한다.
  - canvas 최대 dimension/area를 제한하고, region edge를 `Math.round()`로 정수화한다.
  - 큰 region을 snapshot tile로 나누고 `devicePixelRatio` 변환 후 `drawImage()` 위치를 보정해 gap/line을 줄인다.
- `browser/actors/ScreenshotsComponentChild.sys.mjs`
  - `getFullPageBounds()`가 `scrollMinX`, `scrollMinY`, `scrollWidth`, `scrollHeight`, `devicePixelRatio` 기반 rect를 반환한다.
  - `getVisibleBounds()`는 현재 viewport rect를 같은 구조로 반환한다.
- `browser/components/screenshots/screenshots-buttons.js`
  - visible/full page 버튼이 같은 `takeScreenshot()` 경로에서 type만 다르게 분기한다.

이번 task에서 반영할 점:

- full page bounds는 Firefox와 같은 `left`, `top`, `right`, `bottom`, `width`, `height`, `devicePixelRatio` contract로 정리한다.
- max canvas dimension/area와 tile size는 named policy constant로 둔다.
- stitching 좌표는 fractional DPR/zoom에서 seam이 생기지 않도록 source/destination rect를 정수 pixel 기준으로 snap한다.
- visible capture와 full page capture는 action pipeline을 공유하되 capture backend만 분리한다.

이번 task에서 반영하지 않을 점:

- `drawSnapshot`, `BrowsingContext`, `JSWindowActor`, `ChromeUtils`, `Services`, XPCOM, Firefox preview dialog는 Chrome MV3에서 사용할 수 없으므로 포팅하지 않는다.
- Firefox MPL 코드를 실질적으로 복사하지 않는다. 코드 복사가 필요해지는 경우 구현을 중단하고 `src/firefox-derived/` MPL boundary, `THIRD_PARTY.md`, `NOTICE` 갱신을 별도 승인받는다.

## 잠정 단계

- **Stage 1 — full page capture 모델과 tile/stitch 계약 작성**
  - `full-page-capture`/`stitch-image` 순수 helper와 테스트를 먼저 작성한다.
  - document dimensions, viewport size, scroll target, source crop rect, destination rect, final canvas size contract를 고정한다.
  - 마지막 tile, DPR/zoom-like image ratio, horizontal scroll, max canvas 방어 정책을 검증한다.

- **Stage 2 — capture loop와 overlay 숨김/scroll restoration 구현**
  - content overlay에서 full page capture loop를 구현하고 기존 `captureVisibleTab` message를 재사용하거나 최소 확장한다.
  - capture 중 overlay visibility 숨김, wait-for-paint, scroll 이동, 실패 시 복구, 시작 scroll 위치 restoration을 구현한다.
  - capture loop cancellation/error path를 단위 테스트 가능한 구조로 분리한다.

- **Stage 3 — `전체 페이지 선택` UI 연결과 Copy/Save action 통합**
  - 비활성 placeholder를 실제 버튼으로 전환한다.
  - full page mode 실행 후 Copy/Save 결과와 filename/status metadata를 기존 action pipeline과 통합한다.
  - visible mode, selected mode, full page mode 간 focus/aria 상태와 Firefox식 toolbar UI 회귀를 검증한다.

- **Stage 4 — sticky/fixed/edge fixture와 stitching 품질 회귀 보강**
  - 긴 문서 fixture를 보강하거나 신규 fixture를 추가한다.
  - sticky header, fixed element, large document, partial bottom tile, zoom-like mapping에 대한 자동/수동 smoke 기준을 남긴다.
  - stitching seam이 눈에 띄게 어긋나지 않는지 pixel 또는 dimension 검사를 추가한다.

- **Stage 5 — 문서, 품질 매트릭스, 최종 smoke와 보고**
  - README와 품질 매트릭스를 full page 지원 상태로 갱신한다.
  - 권한 추가가 없음을 검증하고, 제한 사항과 잔여 위험을 최종 보고서에 기록한다.
  - 사용자가 직접 smoke할 체크리스트와 fixture URL을 정리한다.

## 리스크

- **Chrome capture 제약**: `captureVisibleTab()`은 실제 확장 context와 브라우저 rate/permission 제약을 받는다. 순수 helper는 단위 테스트하고 runtime은 fixture smoke로 보완한다.
- **권한 범위 확대 유혹**: debugger 또는 host permission을 쓰면 구현은 쉬워질 수 있지만 AGENTS/MVP 정책과 충돌한다. 필요 시 즉시 중단하고 별도 승인 task로 분리한다.
- **stitching seam**: scroll 위치, DPR, zoom, 마지막 tile crop이 어긋나면 seam이 생긴다. tile plan/source crop/destination rect를 먼저 테스트로 고정한다.
- **sticky/fixed 중복**: scroll stitching 방식은 sticky/fixed element가 여러 tile에 반복될 수 있다. Stage 1에서 정책을 확정하고, 최소 보정 또는 제한 문서화를 명확히 한다.
- **layout shift**: lazy loading, scroll anchoring, animation, sticky header가 capture 중 layout을 바꿀 수 있다. fixture에서 안정 문서를 우선 검증하고 실제 웹 잔여 위험으로 기록한다.
- **큰 canvas 한계**: 매우 긴 페이지는 browser canvas 크기 한계에 걸릴 수 있다. 최대 높이/에러 처리 정책을 Stage 1에서 정한다.

- sticky/fixed element 처리는 Stage 1에서 MVP 정책을 테스트/문서로 먼저 고정한 뒤 구현하는 것
