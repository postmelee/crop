# Task #51 수행계획서 - Chrome Web Store 게시 완료 후 README 설치 안내 갱신

GitHub Issue: [#51](https://github.com/postmelee/crop/issues/51)
마일스톤: M030

## 목적

Chrome Web Store 게시가 완료된 상태에 맞춰 루트 README의 첫 사용자 안내를 공식 설치 경로 중심으로 갱신한다. 사용자는 더 이상 소스 빌드와 압축 해제 로드를 첫 설치 경로로 읽지 않아야 하며, 개발자와 기여자는 기존 source loading 안내를 개발용 fallback으로 계속 확인할 수 있어야 한다.

이번 task는 작업지시자가 제공한 공개 listing URL `https://chromewebstore.google.com/detail/crop/pdmniipgbjdcpnhbkkppodechbehagki`를 README 계열 문서에 반영하고, release status 문구와 설치 섹션이 privacy, permissions, current limits 설명과 충돌하지 않도록 정리한다.

## 배경

이슈 #51은 Chrome Web Store 심사 완료 전까지 `available now`, `published`, `install from Chrome Web Store` 같은 표현을 추가하지 않도록 제한했다. 현재 작업지시자가 Chrome Web Store 게시 완료와 공개 listing URL을 공유했으므로, 이슈의 시작 조건이 충족됐다.

현재 `README.md`, `README.ko.md`, `README.ja.md`, `README.zh-CN.md`는 모두 "Chrome Web Store에 아직 listed되지 않았고 source build/load unpacked를 사용한다"는 release preparation 문구를 포함한다. 게시 완료 후에는 이 문구가 사용자 설치 안내와 충돌하므로, 언어별 README를 같은 게시 상태로 맞춰야 한다.

## 범위

### 포함

- Chrome Web Store 공식 listing URL을 README 설치 안내에 반영한다.
- release status 문구를 게시 완료 상태로 갱신한다.
- 사용자용 Chrome Web Store 설치 안내와 개발자용 source build/load unpacked 안내를 분리한다.
- privacy, permissions, current limits 문구와 설치 안내의 충돌 여부를 확인한다.
- `README.md`, `README.ko.md`, `README.ja.md`, `README.zh-CN.md`의 게시 상태를 같은 기준으로 맞춘다.

### 제외

- Chrome Web Store listing copy 자체 변경
- extension 기능, 권한, privacy policy 본문 변경
- full-page capture, iframe selection, capture backend 동작 변경
- Store 심사 상태 확인 자동화
- 새 공식 문서 루트 생성

## 설계 방향

- 루트 README 4개 언어 파일을 사용자-facing 공식 문서로 보고 같은 상태 문구를 유지한다.
- 각 README 상단의 release status를 "Chrome Web Store에서 설치 가능"한 상태로 바꾸고, 바로 이어지는 설치 섹션에서 공식 URL을 우선 노출한다.
- 기존 `Load from source` 섹션은 삭제하지 않고 개발자용 fallback 또는 development loading 경로로 재배치한다.
- Chrome Web Store 제한 페이지에서는 확장 주입이 막힌다는 current limits 문구는 유지한다. 설치 링크가 Chrome Web Store를 가리켜도 해당 제한과 모순되지 않도록 표현을 분리한다.
- 제품명과 사용자-facing 브랜딩은 `crop`만 사용하고, Mozilla/Firefox 명칭은 기존 attribution 범위 밖으로 확장하지 않는다.
- 권한 설명은 현재 manifest와 README의 권한 표가 일치하는지 확인하되, 이번 task에서 권한을 추가하거나 제거하지 않는다.

## 잠정 단계

- **Stage 1 — README 현황과 게시 URL 기준 확인**
  - 공개 listing URL 접근 가능 여부와 README 4개 언어 파일의 현재 release/install 문구를 확인한다.
  - permissions, privacy, current limits 섹션이 설치 문구와 충돌할 수 있는 지점을 기록한다.
  - 검증 관점은 README 수정 전에 변경 대상과 제외 대상을 고정하는 것이다.
- **Stage 2 — README 설치 안내 갱신**
  - README 4개 언어 파일의 release status와 설치 섹션을 공식 Store 설치 우선 구조로 갱신한다.
  - source build/load unpacked 안내는 개발자용 fallback으로 유지한다.
  - 검증 관점은 각 언어 README가 같은 게시 상태, 같은 Store URL, 같은 권한/개인정보 경계를 유지하는 것이다.
- **Stage 3 — 최종 검증과 보고**
  - README grep, 링크 확인, diff check를 수행하고 최종 보고서를 작성한다.
  - 변경하지 않은 항목과 다국어 README 변경 근거를 보고서에 남긴다.
  - 검증 관점은 수용 기준이 모두 충족됐고 PR로 보낼 문서 상태가 정리됐는지 확인하는 것이다.

## 검증 계획

### 단계별 검증

- Stage 1
  - Chrome Web Store 공개 listing URL 접근 확인
  - `rg -n "Release status|Chrome Web Store|listed|Load from source|Privacy Policy|Permissions|Current Limits" README.md README.ko.md README.ja.md README.zh-CN.md`
  - `rg -n "debugger|<all_urls>|host_permissions|downloads|activeTab|scripting|clipboardWrite" manifest.json README.md README.ko.md README.ja.md README.zh-CN.md`
- Stage 2
  - `rg -n "Chrome Web Store|pdmniipgbjdcpnhbkkppodechbehagki|load unpacked|Load unpacked|Privacy Policy|Permissions|Current Limits" README.md README.ko.md README.ja.md README.zh-CN.md`
  - `git diff -- README.md README.ko.md README.ja.md README.zh-CN.md`
- Stage 3
  - `git diff --check`
  - `git status --short`
  - `rg -n "pdmniipgbjdcpnhbkkppodechbehagki|debugger|<all_urls>|host_permissions" README.md README.ko.md README.ja.md README.zh-CN.md`

### 통합 검증

- README의 첫 사용자 안내가 Chrome Web Store 공식 설치 경로를 우선한다.
- source build/load unpacked 안내가 개발자용 경로로 남아 있다.
- Store listing URL `https://chromewebstore.google.com/detail/crop/pdmniipgbjdcpnhbkkppodechbehagki`가 README 계열 문서에 반영된다.
- privacy, permissions, current limits 문구가 새 설치 안내와 충돌하지 않는다.
- 다국어 README 4개가 같은 게시 상태를 설명한다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **Chrome Web Store 동적 페이지 확인 한계**: Chrome Web Store는 지역, 로그인 상태, 동적 렌더링에 따라 접근 결과가 달라질 수 있다. 작업지시자가 제공한 공식 URL을 기준으로 하되, 접근 확인 결과와 한계는 단계 보고서에 기록한다.
- **다국어 문구 불일치**: 4개 README를 동시에 수정하므로 언어별 설치 안내와 제한사항이 어긋날 수 있다. Stage 2에서 동일 URL과 같은 상태 전환 표현을 grep으로 확인한다.
- **권한 설명과 manifest 불일치**: README 권한 표가 manifest와 어긋나면 사용자-facing 신뢰 문제가 생긴다. 이번 task에서는 권한 변경 없이 문서 일치 여부만 확인하고, 기능/권한 변경이 필요하면 별도 이슈로 분리한다.
