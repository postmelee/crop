# Task #13 수행계획서

GitHub Issue: [#13](https://github.com/postmelee/crop/issues/13)
마일스톤: M020

## 목적

선택이 확정된 rectangle을 사용자가 다시 조정할 수 있도록 Firefox Screenshots식 resize/move controls를 구현한다. 현재 crop은 hover/click/drag로 selected rectangle을 만들고 Copy/Save/Cancel을 표시하지만, 선택 후 크기나 위치를 바꾸려면 selection을 해제하고 다시 선택해야 한다.

이번 task는 #12 edge auto-scroll 이후의 M020 Firefox parity 후속 작업이다. selected 상태에서 모서리/변 handle drag, 내부 drag move, keyboard 기반 조정을 제공하고, action box가 viewport 경계에서도 접근 가능한 위치에 유지되도록 보정한다.

## 배경

#5에서 page-coordinate selection과 selected outside reset을 구현했고, #8에서 Phase 6 smoke로 visible viewport capture와 선택 경계 동작을 안정화했다. #12에서는 drag selection 중 edge auto-scroll을 추가해 문서 좌표 기반 selection 흐름을 확장했다. 남은 큰 UX 차이는 selected 상태에서 영역을 직접 수정할 수 없다는 점이다.

Issue #13은 Firefox 원본의 `ScreenshotsOverlayChild.sys.mjs`에 있는 `resizingDrag()`, `selectedDragStart()`, `drawSelectionContainer()` 흐름과 `overlay.css`의 selected region/action box 스타일을 참고 대상으로 둔다. 다만 Chrome MV3 content script 구조, Shadow DOM overlay, visible viewport capture 정책을 유지해야 하므로 privileged API나 새 권한을 추가하지 않는다.

## 설계 방향

- 기존 overlay state의 page-coordinate 원칙을 유지한다. selected, moving, resizing 중인 rectangle은 page 좌표로 저장하고 render 직전에 viewport 좌표로 투영한다.
- 초기 drag selection과 selected 상태 조정은 상태를 분리한다. 기존 `draggingReady`/`dragging`은 새 영역 생성에 유지하고, selected 조정은 `moving`/`resizing` 또는 동등한 명시 상태로 모델링한다.
- resize/move 계산은 작은 순수 helper로 분리한다. helper는 handle 방향, 시작 rect, 시작 pointer, 현재 pointer, viewport/page 제한 정책을 입력받아 normalized `PageRect`를 반환한다.
- 최소 selection 크기를 둬 handle drag가 0px 또는 뒤집힌 rectangle을 만들지 않도록 한다. 역방향 resize 허용 여부는 helper 테스트로 명확히 고정한다.
- 선택 영역 내부와 handle은 Shadow DOM 요소로 렌더링하되, capture 직전 overlay 전체 숨김 정책은 유지해 최종 PNG에 UI가 포함되지 않도록 한다.
- pointer hit-test는 action button, resize handle, selected highlight 내부, document outside 순서로 판정한다. selected 내부 조정 이벤트가 outside reset으로 이어지지 않도록 document click suppression을 함께 정리한다.
- keyboard 조정은 selected 상태에서만 처리한다. 기본 Arrow는 selection move, Shift+Arrow는 더 큰 step move, Alt/Option+Arrow는 해당 방향 edge resize, Alt/Option+Shift+Arrow는 더 큰 step resize로 검토한다. 플랫폼 충돌이 있으면 Stage 1에서 대안을 확정하고 구현계획서에 기록한다.
- action box는 현재 `positioning.ts`의 flip/clamp helper를 확장해 Firefox식 dark rounded toolbar 시각과 placement 규칙을 테스트 가능한 값으로 고정한다.
- MVP 권한 정책을 유지한다. `manifest.json`에 새 권한을 추가하지 않는다.

## 리스크

- **pointer/click sequence 충돌**: selected 내부 drag나 handle pointerdown이 기존 outside reset/click select 흐름과 충돌할 수 있다. hit-test 우선순위와 document click suppression을 Stage 2에서 테스트로 고정한다.
- **좌표 drift**: page 좌표와 viewport 좌표를 섞으면 scroll 이후 move/resize 결과가 밀릴 수 있다. state 저장은 page 좌표, render는 viewport 투영 원칙을 유지한다.
- **작은 selection 조작성**: 작은 rectangle에서 8방향 handle이 겹치거나 action box와 충돌할 수 있다. 최소 크기, handle hit area, action box placement를 함께 검증한다.
- **keyboard shortcut 충돌**: 브라우저/페이지 기본 키 동작과 조정 키가 충돌할 수 있다. selected 상태에서만 제한적으로 preventDefault하고, action button focus 사용성을 침해하지 않도록 한다.
- **시각 parity 범위 초과**: Firefox식 action box 보정이 unrelated restyle로 번질 수 있다. selected controls와 action toolbar에만 변경을 제한한다.
- **자동화 한계**: 확장 Shadow DOM pointer drag와 screenshot 비교는 자동화가 불안정할 수 있다. helper/state 단위 테스트를 우선 고정하고, 필요한 경우 작업지시자 수동 smoke 지침을 제공한다.
