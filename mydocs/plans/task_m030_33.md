# Task #33 수행계획서 - 확장 UI 사용자-facing 텍스트 다국어화

GitHub Issue: [#33](https://github.com/postmelee/crop/issues/33)
마일스톤: M030

## 목적

Chrome 확장 `crop`의 manifest와 오버레이 UI에 하드코딩된 사용자-facing 문자열을 Chrome Extension 표준 i18n 구조로 전환한다. README 다국어화와 같은 언어 축인 English, Korean, Japanese, Simplified Chinese를 기본 지원하고, Chrome UI locale에 따라 버튼, 안내 문구, 토스트, 접근성 label/title이 자연스럽게 표시되도록 한다.

이번 task는 사용자에게 노출되는 텍스트를 다국어화하는 작업이며, MVP 권한과 캡처/Copy/Save 동작은 유지한다.

## 배경

README family는 이미 다국어 문서로 정리됐지만 실제 확장 UI는 `복사`, `저장`, `취소`, 안내 프롬프트, 복사 완료 토스트, preview dialog aria label 등 문자열이 `src/content/overlay/`에 직접 박혀 있다. `manifest.json`의 description과 command description도 아직 고정 문자열이다.

Chrome 공식 i18n 방식은 `_locales/{locale}/messages.json`, manifest의 `__MSG_*__`, JavaScript의 `chrome.i18n.getMessage()`를 사용한다. content script는 `chrome.i18n`에 직접 접근할 수 있으므로 Shadow DOM 오버레이도 별도 권한 추가 없이 지역화할 수 있다.

## 범위

### 포함

- `manifest.json`의 name, description, action title, command description i18n 적용
- `_locales/en`, `_locales/ko`, `_locales/ja`, `_locales/zh_CN` 메시지 파일 추가
- `src/content/overlay/`의 버튼, 프롬프트, 토스트, preview, status, aria-label/title, resize handle label 지역화
- Vite build 결과에 `_locales`가 포함되도록 빌드 설정 보강
- locale key와 UI 문자열 회귀를 확인하는 테스트 갱신

### 제외

- Chrome Web Store listing 문구, 스크린샷, 프로모션 자산 지역화
- 사용자가 확장 내부에서 직접 언어를 선택하는 설정 UI
- README 추가 번역 작업
- 내부 개발자용 console warning/error 전체 번역
- 신규 Chrome extension 권한 추가

## 설계 방향

- Chrome Extension 표준 i18n을 우선 사용한다. manifest는 `__MSG_key__`, runtime UI는 `chrome.i18n.getMessage("key")`로 문자열을 조회한다.
- 기본 locale은 `en`으로 둔다. 모든 key는 `_locales/en/messages.json`에 정의하고, `ko`, `ja`, `zh_CN`은 같은 key set을 유지한다.
- content script 단위 테스트와 브라우저 외 실행 환경을 위해 작은 i18n wrapper를 둔다. Chrome API가 없는 테스트 환경에서는 English fallback map 또는 명시 주입 가능한 lookup을 사용한다.
- 사용자-facing 문자열만 다국어화한다. 로그, 내부 dataset 값, 테스트 marker, capture action id 같은 내부 계약 문자열은 번역하지 않는다.
- `zh-CN` README와 달리 Chrome locale 디렉터리는 공식 locale code인 `zh_CN`을 사용한다.
- `_locales`는 extension package 루트에 있어야 하므로 Vite build 시 `dist/_locales/...`로 복사한다.

## 문서 위치 판단

이번 task는 공식 제품/사용자 문서를 새로 만들거나 수정하지 않는다. `_locales/*/messages.json`은 사용자-facing 번역 리소스지만 Chrome extension 런타임 패키지 자산이며 문서 루트가 아니다. 작업 산출물 문서는 하이퍼-워터폴 규칙에 따라 `mydocs/`에 둔다.

## 리스크

- **테스트 환경의 Chrome API 부재**: Vitest/jsdom 환경에는 `chrome.i18n`이 없을 수 있다. 작은 wrapper와 fallback/주입 지점을 둬 테스트 안정성을 확보한다.
- **locale key 누락**: default locale fallback이 있더라도 특정 언어 누락은 품질 저하다. key parity 테스트를 추가한다.
- **빌드 산출물 누락**: `_locales`가 source에는 있어도 `dist`에 빠지면 unpacked extension에서 지역화가 동작하지 않는다. build 검증에 `dist/_locales` 확인을 포함한다.
- **하드코딩 문자열 잔존**: 사용자-facing 문자열과 내부 계약 문자열을 구분해야 한다. grep과 테스트를 병행하되 내부 action id/status id는 번역하지 않는다.
- **버튼 폭 변화**: 번역 문자열 길이 차이로 action box 폭이나 preview footer가 변할 수 있다. 기존 CSS와 positioning 테스트를 유지하고 필요 시 문자열이 넘치지 않도록 조정한다.

## 승인 요청 사항

- 기본 locale을 `en`으로 두고 `ko`, `ja`, `zh_CN`을 같은 key set으로 제공하는 방향
- 사용자-facing UI 문자열만 번역하고 내부 로그/상태 id는 제외하는 방향
- `_locales`를 extension root의 런타임 리소스로 추가하고 Vite build가 `dist/_locales`로 복사하도록 하는 방향
- 잠정 Stage 1~3 구성
