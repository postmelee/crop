# Task M020 #68 수행계획서

GitHub Issue: [#68](https://github.com/postmelee/crop/issues/68)
마일스톤: M020

## 목적

macOS `Show scroll bars: Always` 환경에서 요소 선택 후 Copy/Save 시 오른쪽에 불필요한 여백이 포함되는 문제를 보정한다.

선택 overlay와 DOM hit-test는 실제 콘텐츠 viewport 기준을 유지하되, `chrome.tabs.captureVisibleTab()`으로 얻은 비트맵을 source pixel로 변환할 때는 캡처 이미지가 기준으로 삼는 visual viewport 크기를 사용하도록 분리한다. 결과적으로 스크롤바 표시 정책과 관계없이 선택 영역과 출력 PNG의 좌표가 일치해야 한다.

## 배경

#68 제보 이미지의 Crop 출력은 `726 x 405`이고 원본 이미지 비율상 기대 폭은 약 `720px`이다. 재현 확인 결과 macOS 스크롤바 표시 설정을 `Always`로 바꾸면 같은 오른쪽 여백이 발생했다.

현행 단일 viewport capture 경로는 `captureVisibleTab()` 결과 이미지의 natural size를 `documentElement.clientWidth/clientHeight`로 나누어 scale을 계산한다. macOS classic scrollbar 환경에서는 캡처 비트맵 폭이 `window.innerWidth` 계열이고 DOM 콘텐츠 폭은 `documentElement.clientWidth` 계열이라 약 12px 차이가 생긴다. 이 차이가 선택 폭에 비례해 source crop rect를 과대 계산한다.

관련 경로:

- `src/content/overlay/crop-overlay.ts`의 selected/visible viewport capture
- `src/content/overlay/full-page-capture.ts`의 full-page 및 viewport 밖 selected tile planning
- `src/shared/crop-image.ts`, `src/shared/stitch-image.ts`의 source pixel mapping helper

## 범위

### 포함

- macOS Always scroll bars처럼 `window.innerWidth`와 `documentElement.clientWidth`가 다른 환경의 source crop mapping 보정
- selected visible capture와 visible viewport preview capture 보정
- full-page capture와 viewport 밖 selected stitching 경로의 capture viewport size 계약 점검 및 필요 보정
- 관련 unit/regression test 추가
- 권한 범위와 MV3 capture backend 유지 검증

### 제외

- 픽셀 색상 기반 흰 여백 자동 trim
- `debugger`, `<all_urls>`, broad host permission 추가
- full-page stitching 품질 전반 재설계
- 이미지 wrapper 자동 선택 휴리스틱 변경
- 사용자-facing 문서 신규 작성

## 설계 방향

- DOM 선택, hover, overlay 배치는 계속 콘텐츠 viewport(`documentElement.clientWidth/clientHeight`) 기준으로 둔다.
- 캡처 비트맵 source pixel mapping에는 `window.innerWidth/innerHeight` 기반의 capture viewport size를 별도로 전달한다.
- `cropPngDataUrl()` 자체는 순수 변환 helper로 유지하고, caller가 올바른 viewport size를 전달하는 구조를 우선한다.
- full-page/selected stitching은 tile segment 크기와 capture source mapping 크기를 혼동하지 않도록 계약을 분리한다.
- 오른쪽 여백을 결과 이미지에서 색상으로 잘라내는 방식은 정상 흰 배경 콘텐츠를 훼손할 수 있으므로 사용하지 않는다.
- Chrome MV3 MVP 권한과 `captureVisibleTab()` 기반 backend를 유지한다.

## 리스크

- **viewport 기준 혼동**: tile planning까지 `innerWidth`로 바꾸면 콘텐츠 scroll segment가 바뀔 수 있다. DOM 콘텐츠 viewport와 capture source viewport를 분리해 대응한다.
- **full-page 회귀**: full-page stitching은 destination edge alignment가 중요하다. source mapping 보정과 destination mapping을 분리하고 기존 alignment 테스트를 유지한다.
- **환경 의존 수동 검증**: Always scroll bars 설정은 OS/Chrome 조합에 의존한다. 자동 테스트로 수식 회귀를 고정하고, 작업지시자 수동 smoke를 최종 확인으로 둔다.
- **흰 배경 오판**: 픽셀 trim 방식은 정상 흰 콘텐츠를 잘라낼 수 있다. 좌표 계약 보정만 수행한다.

- 흰 픽셀 trim, 권한 확대, 이미지 선택 휴리스틱 변경을 이번 task 범위에서 제외하는 것
