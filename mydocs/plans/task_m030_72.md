# Task #72 수행계획서 - 다음 릴리즈 후보 정리와 devel -> main 승격 준비

GitHub Issue: [#72](https://github.com/postmelee/crop/issues/72)
마일스톤: M030

## 목적

다음 `crop` 릴리즈를 개시하기 전에 `v0.1.0` 이후 `main`과 `devel`에 누적된 변경을 release 후보로 정리한다. 사용자-facing 변경점과 maintainer 검증 기록을 분리한 GitHub Release body 후보를 만들고, `devel -> main` release PR을 열 수 있는 검증 기준과 PR 본문 후보를 준비한다.

이번 task는 Chrome Web Store 새 제출을 수행하지 않는다. 작업지시자가 Store 제출을 직접 진행할 수 있도록 package 검증 결과, release asset 후보, checksum, privacy URL 후보, 제출 중단 조건만 산출물에 남긴다.

## 배경

현재 공개 release와 tag 기준은 `v0.1.0`이다. 이후 `main`에는 README badge/asset 계열 보강이 있고, `devel`에는 #66과 #68의 캡처 안정화 보정이 추가로 누적되어 있다. `package.json`과 `manifest.json`의 version은 아직 `0.1.0`이므로 새 release version을 `v0.1.1`로 둘지 다른 버전으로 둘지 먼저 승인받아야 한다.

#46은 release pipeline과 package 검증 기준을 정했고, #50은 GitHub Release note 템플릿을 만들었으며, #54는 기존 `v0.1.0` Release body를 표준 구조로 보정했다. 이번 task는 그 절차를 실제 다음 release 후보에 적용하는 운영 작업이다.

## 범위

### 포함

- `v0.1.0..origin/main`, `v0.1.0..origin/devel`, `origin/main..origin/devel` 변경사항 수집
- 포함 PR/Issue 목록과 사용자-facing 변경점 정리
- release version 후보 결정과 `package.json`, `manifest.json` version bump 필요 여부 판단
- 승인된 version 기준 release body 후보 작성
- `devel -> main` release PR 본문 후보 작성
- `npm run build`, `npm run typecheck`, `npm test`, `npm run package:cws`, `npm run verify:cws`
- ZIP contents, forbidden entry, manifest permission, checksum 확인
- `main` 또는 release tag 기준 privacy URL 후보와 Chrome Web Store 제출 중단 조건 정리
- 작업 단계 보고서, 최종 보고서, 오늘할일 갱신

### 제외

- Chrome Web Store package upload 또는 `Submit for review`
- 새 기능 추가 또는 별도 bug fix 구현
- `debugger`, `<all_urls>`, broad `host_permissions`, `tabs` 권한 추가
- release 자동화 workflow 또는 `.github/release.yml` 도입
- 기존 `v0.1.0` Release body 수정
- 작업지시자 별도 승인 전 `main` merge, tag 생성, GitHub Release 생성

## 설계 방향

- release 후보 version은 Stage 1에서 변경 성격을 대조한 뒤 제안한다. 현재 관찰상 기능 안정화와 문서/운영 보강 중심이므로 `v0.1.1`을 우선 후보로 검토하되, 최종 승인 전에는 version 파일을 바꾸지 않는다.
- release body 후보는 `mydocs/_templates/github_release_note.md` 구조를 따른다. 사용자 안내와 developer 검증 기록을 분리하고, Chrome Web Store 상태는 작업지시자가 직접 제출한다는 전제에 맞춰 `not submitted` 또는 승인된 표현으로 고정한다.
- release body 후보와 release PR 본문 후보의 task-specific 근거는 `mydocs/tech/`에 둔다. 실제 GitHub Release body의 진실 원천은 GitHub Release가 되지만, 게시 전 후보와 검증값은 작업 산출물로 남긴다.
- release PR은 일반 task PR과 분리되는 `devel -> main` 흐름이다. 이번 task의 일반 PR은 #72 작업 산출물을 `devel`에 반영하는 데 사용하고, release PR 생성/merge/tag는 별도 승인 지점으로 둔다.
- `main` 전용 README asset 변경과 `devel` 전용 캡처 보정이 모두 다음 release 후보에 포함될 수 있으므로, tree diff만 보지 않고 tag 기준 변경과 branch 간 차이를 함께 검토한다.

## 잠정 단계

- **Stage 1 — release 후보 범위와 version 판단**
  - 원격 branch/tag 상태, `v0.1.0` 이후 변경, 포함 PR/Issue, 현재 version 값을 확인한다.
  - `v0.1.1` 또는 다른 후보 version의 근거와 승인 지점을 정리한다.
  - 검증 관점은 release 후보 범위가 `main` 전용 변경과 `devel` 누적 변경을 모두 빠짐없이 다루는지 확인하는 것이다.
- **Stage 2 — version bump와 release candidate 초안 작성**
  - 승인된 version 기준으로 필요한 version 파일을 갱신한다.
  - GitHub Release body 후보와 release PR 본문 후보를 작성한다.
  - 검증 관점은 release note placeholder가 남지 않고 사용자 안내와 developer 검증 기록이 분리되는지 확인하는 것이다.
- **Stage 3 — release package 검증**
  - build/typecheck/test/package/verify를 실행한다.
  - ZIP contents, forbidden entry, manifest permission, file size, SHA-256 checksum을 기록한다.
  - 검증 관점은 Chrome Web Store 제출용 package 후보가 release 기준으로 재현 가능한지 확인하는 것이다.
- **Stage 4 — 최종 release PR 준비 보고**
  - 최종 보고서에 release version, release body 후보, release PR 본문 후보, package checksum, privacy URL 후보, 남은 승인 항목을 정리한다.
  - `main` merge, tag/GitHub Release 생성, Chrome Web Store 제출은 수행하지 않고 별도 승인 요청으로 남긴다.

## 검증 계획

### 단계별 검증

- Stage 1
  - `git fetch --all --tags --prune`
  - `git status --short --branch`
  - `git tag --sort=-creatordate`
  - `git log --oneline v0.1.0..origin/main`
  - `git log --oneline v0.1.0..origin/devel`
  - `git diff --name-status origin/main..origin/devel`
  - `node -p "require('./package.json').version"`
  - `rg -n '"version"|debugger|<all_urls>|host_permissions|tabs' package.json manifest.json src tests`
- Stage 2
  - `rg -n "v0.1.1|Release|Chrome Web Store|checksum|privacy|verification|placeholder|해당 없음" mydocs/tech/task_m030_72_release_candidate.md mydocs/plans/task_m030_72_impl.md`
  - `rg -n '"version": "0.1.1"' package.json manifest.json` 또는 승인된 version 기준 equivalent grep
  - `git diff --check`
- Stage 3
  - `npm run build`
  - `npm run typecheck`
  - `npm test`
  - `npm run package:cws`
  - `npm run verify:cws`
  - `unzip -l /tmp/crop-{version}-cws.zip`
  - `unzip -Z1 /tmp/crop-{version}-cws.zip`
  - `shasum -a 256 /tmp/crop-{version}-cws.zip`
- Stage 4
  - `rg -n "Release PR|GitHub Release|Submit for review|Chrome Web Store|checksum|SHA-256|privacy|승인" mydocs/tech mydocs/working mydocs/report`
  - `git diff --check`

### 통합 검증

- release version이 승인되어 있고 version 파일이 그 기준과 일치한다.
- release body 후보에 placeholder가 남아 있지 않다.
- 사용자 안내에는 실제 사용자-facing 변경과 known limitations만 들어간다.
- developer 검증 기록에는 포함 PR/Issue, commit 후보, asset 후보, checksum, verification 결과가 들어간다.
- `devel -> main` release PR 전에 필요한 로컬 검증이 통과한다.
- `debugger`, `<all_urls>`, broad `host_permissions`, `tabs` 권한이 추가되지 않는다.
- Chrome Web Store package upload와 `Submit for review`를 수행하지 않았음이 최종 보고서에 명시된다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **version 판단 오류**: `v0.1.0` 이후 변경이 patch/minor 중 어디에 해당하는지 애매할 수 있다. Stage 1에서 변경 성격과 작업지시자 승인 지점을 분리한다.
- **branch 기준 혼선**: `main`과 `devel`이 서로 다른 변경을 가지고 있다. tag 기준 변경과 branch 간 차이를 모두 확인해 release note 누락과 asset 손실 오해를 줄인다.
- **release PR과 task PR 경계 혼동**: #72 task PR은 산출물을 `devel`에 반영하는 일반 PR이고, `devel -> main`은 별도 release PR이다. 수행계획서와 최종 보고서에 승인 경계를 명시한다.
- **package artifact 불일치**: version bump 후 ZIP 이름, size, checksum이 바뀐다. Stage 3에서 fresh package와 checksum을 release 후보 기준으로 다시 고정한다.
- **Chrome Web Store 상태 표현 불일치**: 작업지시자가 Store 제출을 직접 진행하므로 release body의 Store 상태 표현은 실제 제출 전 상태와 맞아야 한다. Stage 2/4에서 `not submitted`, `draft only` 등 승인된 표현으로 확정한다.

## 승인 요청 사항

- 위 범위와 Stage 1~4 구성으로 #72를 진행한다.
- release version은 Stage 1에서 `v0.1.1`을 우선 후보로 검토하되, 작업지시자 승인 전 version 파일을 변경하지 않는다.
- release body 후보와 release PR 본문 후보는 `mydocs/tech/task_m030_72_release_candidate.md`에 보관한다.
- Chrome Web Store package upload와 `Submit for review`는 이번 task에서 제외한다.
- `main` merge, tag 생성, GitHub Release 생성은 별도 명시 승인 전 수행하지 않는다.

승인되면 `task_m030_72_impl.md`에서 단계별 산출물, 검증 명령, 커밋 메시지를 구체화한다.
