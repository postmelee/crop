# Task #28 수행계획서

GitHub Issue: [#28](https://github.com/postmelee/crop/issues/28)
마일스톤: M030

## 목적

`crop` 저장소의 공개 첫 화면을 Chrome extension 사용자 관점으로 재정리한다. 루트 `README.md`는 English를 기본 언어로 두고, 기능 소개, 설치/로드 방법, 사용 흐름, 권한 사용 이유, privacy stance, 제한사항, license/attribution을 현재 구현과 일치하게 설명한다.

또한 Korean, Chinese, Japanese README를 링크로 접근할 수 있는 구조를 만든다. 이 구조는 Chrome Web Store 설명, 브랜딩 자산, Community Standards 문서에서 같은 공개 문구를 재사용할 수 있는 기준점이 되어야 한다.

## 배경

M030 #9 배포 준비 과정에서 현재 `README.md`는 개발 상태와 내부 검증 기록이 길게 누적되어 있다. 이는 작업자에게는 유용하지만, 처음 방문한 사용자가 `crop`이 무엇을 하는 확장인지, 어떤 권한을 쓰는지, 캡처 이미지가 어디에서 처리되는지, 어떤 페이지/상황이 제한되는지 빠르게 이해하기 어렵다.

#28은 배포 전 공개 문구의 기준을 먼저 세우는 작업이다. #29 Community Standards 보강과 추후 브랜딩 자산 작업은 이 README의 기능 설명, 권한 설명, privacy/local processing 설명을 참조하게 된다.

참고 기준은 #28 이슈 본문, #9 배포 준비 이슈, 루트 `README.md`, `manifest.json`, 라이선스/출처 문서(`LICENSE`, `LICENSE-MPL-2.0`, `NOTICE`, `THIRD_PARTY.md`), 원본 제품 계획(`/Users/melee/Downloads/crop_development_plan_prompt.md`)이다.

## 범위

### 포함

- 루트 `README.md`를 English 사용자-facing entrypoint로 재구성
- 기능 소개, quick start, Chrome unpacked loading, basic usage, permissions, privacy/local processing, limitations, license/attribution 섹션 정리
- Korean, Chinese, Japanese README 링크 구조 추가
- `README.ko.md`, `README.zh-CN.md`, `README.ja.md` 신규 작성
- 기존 개발자용 상세 상태 기록 중 루트 README에 남길 내용과 축약할 내용 판단
- Chrome Web Store와 브랜딩 자산에 재사용 가능한 공개 문구 초안 정리

### 제외

- Chrome Web Store 실제 제출
- 앱 아이콘, 스크린샷, 기능 소개 동영상 제작 또는 반영
- 개인정보 처리방침 전문 확정
- 기능 구현 변경
- `debugger`, `<all_urls>` 권한 추가
- Mozilla/Firefox 명칭을 제품명, 확장명, 아이콘, 제휴 암시 문구로 사용하는 변경
- 공식 문서 사이트, GitHub Pages, `docs/` 루트 신설

## 설계 방향

- 루트 `README.md`의 첫 화면은 `crop`의 사용자 가치, 현재 지원 기능, 설치/사용 시작점이 바로 보이도록 구성한다.
- 언어 전환 링크는 README 상단에 둔다. English 루트 README에서 Korean, Chinese, Japanese README로 이동하고, 각 다국어 README에서도 English 루트로 돌아갈 수 있게 한다.
- Chinese README 파일명은 공개 저장소에서 일반적인 Simplified Chinese 표기인 `README.zh-CN.md`로 둔다. Traditional Chinese는 이번 task 범위에서 제외하고, 필요하면 별도 이슈로 분리한다.
- 다국어 README는 루트 README와 같은 정보 구조를 유지하되, 번역 과정에서 기능 범위를 확대하거나 약속하지 않는다.
- 현재 manifest 권한(`activeTab`, `scripting`, `clipboardWrite`, `downloads`)과 실제 동작을 README 권한 설명의 진실 원천으로 삼는다.
- privacy 문구는 서버 전송 없음, 로컬 처리, telemetry 없음, 사용자가 Copy/Save를 실행할 때만 이미지가 클립보드 또는 다운로드로 나간다는 범위로 제한한다.
- Mozilla/Firefox는 upstream material, license, technical inspiration 맥락에서만 언급한다. 제품명, 확장명, 제휴/보증/공식 호환 암시로 사용하지 않는다.
- 개발자용 긴 상태 목록은 루트 README에서 제거하거나 짧은 Development 섹션으로 축약한다. 상세 작업 이력은 기존 `mydocs/` 산출물과 GitHub Issues/PRs를 기준으로 남긴다.

## 잠정 단계

- **Stage 1 — README 정보 구조와 공개 문구 기준 정리**
  - 현재 README, issue #28/#9, manifest 권한, license/attribution 문서를 기준으로 사용자-facing README 구조를 확정한다.
  - 루트 README에 남길 개발자 정보와 제거/축약할 내부 상태 기록을 구분한다.
  - 검증 관점: README 구조가 사용자, 설치 검토자, 기여자 관점에서 과밀하지 않은지 확인한다.

- **Stage 2 — English README 재작성**
  - 루트 `README.md`를 English 기본 문서로 재작성한다.
  - 기능, 설치, 사용법, 권한, privacy/local processing, limitations, license/attribution을 현재 구현과 맞춘다.
  - 검증 관점: `manifest.json` 권한과 README 권한 설명이 일치하고, 브랜딩 금지 표현이 없는지 확인한다.

- **Stage 3 — 다국어 README 작성과 링크 구조 연결**
  - `README.ko.md`, `README.zh-CN.md`, `README.ja.md`를 루트 README와 같은 정보 구조로 작성한다.
  - 모든 README 상단에 언어 전환 링크를 두고, 링크 경로가 실제 파일과 일치하는지 확인한다.
  - 검증 관점: 번역문이 원문 기능 범위를 확대하지 않고, 각 언어 문서가 같은 제한사항과 attribution을 포함하는지 확인한다.

- **Stage 4 — 문서 검증, 단계 보고, 최종 보고**
  - README 링크, 권한 문구, privacy 문구, license/attribution 문구, branding disclaimer를 통합 검토한다.
  - 문서 변경 task이므로 소스 build는 기본 검증에서 제외하되, 파일 구조나 package/manifest를 건드리면 필요한 명령을 추가한다.
  - 검증 관점: `git diff --check`, 링크 경로 확인, `manifest.json` 비교, 브랜딩 금지 표현 확인이 통과해야 한다.

## 리스크

- **기능 과장**: README를 사용자 문서로 정리하면서 아직 지원하지 않는 Store 배포, 서버 동기화, cross-origin iframe 내부 capture, Chrome 제한 페이지 지원을 암시할 수 있다. 현재 구현과 limitations를 함께 검토해 막는다.
- **권한 설명 불일치**: README 권한 설명이 `manifest.json`과 어긋나면 Chrome Web Store 제출 문구와도 흔들린다. Stage 2와 Stage 4에서 manifest와 직접 비교한다.
- **브랜딩 오해**: Firefox Screenshots UX reference를 설명하다가 Mozilla/Firefox 공식 제품처럼 보일 수 있다. attribution/license 맥락 외 표현을 제거하고 disclaimer를 유지한다.
- **번역 drift**: 다국어 README가 English 원문과 다른 기능 약속을 할 수 있다. 같은 섹션 구조와 제한사항을 유지하고, 링크/권한/attribution 항목을 언어별로 대조한다.
- **문서 위치 확장**: README 작업 중 `docs/`나 별도 문서 사이트가 필요해 보일 수 있다. 이번 task에서는 루트 README family로 제한하고, 공식 문서 사이트는 별도 task 후보로 분리한다.
