# Task #46 수행계획서 - 릴리즈 파이프라인과 PR CI/제출 지침 표준화

GitHub Issue: [#46](https://github.com/postmelee/crop/issues/46)
마일스톤: M030

## 목적

첫 `main` 릴리즈와 Chrome Web Store 제출 전에 반복 가능한 release pipeline, PR CI, package artifact 검증, 제출 runbook을 표준화한다. #37에서 Dashboard 입력값은 정리됐지만, 실제 릴리즈와 제출을 지속적으로 반복하려면 `devel -> main`, tag/GitHub Release, Store 제출용 ZIP 생성, CI 검증 책임을 하나의 절차로 고정해야 한다.

이번 task는 운영 기준과 자동 검증 기반을 만드는 작업이다. 실제 Chrome Web Store 제출, Store asset 제작, `main` 릴리즈 PR 생성은 이 task에서 수행하지 않는다.

## 배경

#37 PR #45에서 Chrome Web Store Dashboard 입력값과 제출 체크리스트를 확정했다. PR 게시 직전 package 검증 중 `dist/.DS_Store`가 제출 ZIP에 포함될 수 있는 리스크가 발견되어 Stage 4.2에서 one-off 명령을 보정했다. 이 발견은 제출용 artifact 생성과 검증을 사람이 매번 임시 명령으로 수행하면 같은 문제가 재발할 수 있음을 보여준다.

현재 저장소에는 `npm run build`, `npm run typecheck`, `npm test` 스크립트가 있고, `.github/workflows/`는 아직 없다. `mydocs/manual/git_workflow_guide.md`와 `mydocs/manual/release_update_protocol.md`는 브랜치와 release PR의 큰 흐름을 다루지만, `crop`의 Chrome Web Store 제출 전 package artifact, privacy URL 기준, Dashboard 제출 타이밍, CI pass/fail 기준을 한곳에서 따라 할 수 있는 runbook은 아직 없다.

## 설계 방향

- 운영 절차는 반복 사용 가능해야 하므로, 이슈별 결과 보고서에만 묻히지 않게 별도 지침 문서로 둔다.
- `mydocs/manual/`에 둘 문서는 특정 #46 사건의 로그가 아니라 반복 가능한 release/CI/runbook 절차만 담는다.
- #46의 조사 과정, 대안 비교, 실제 검증 결과는 `mydocs/tech/`와 단계 보고서에 둔다.
- CI는 현재 프로젝트 스크립트(`build`, `typecheck`, `test`)를 우선 사용한다.
- package artifact 생성은 `dist/`를 ZIP root로 삼고, `.DS_Store`, `__MACOSX`, `node_modules`, `mydocs`, repository root 문서를 제외하는 검증을 표준으로 삼는다.
- GitHub Actions workflow를 추가한다면 PR 대상은 `devel`, release PR 대상은 `main` 기준으로 동작하게 하고, artifact upload는 필요 최소 범위로 둔다.
- Chrome Web Store 제출은 `main` 반영과 release/tag 이후에만 진행한다는 #37 판단을 유지한다.

## 잠정 단계

- **Stage 1 — 현행 release/CI 경계와 요구사항 조사**
  - 현재 `.github/`, `package.json`, #37 산출물, release 관련 manual을 대조한다.
  - GitHub Actions와 artifact 검증에 필요한 공식 문서 또는 primary source를 확인한다.
  - CI를 실제로 추가할지, 문서 기준만 먼저 둘지 판단한다.
- **Stage 2 — package artifact 생성/검증 표준화**
  - 제출 ZIP 생성/검증 절차를 one-off 명령에서 반복 가능한 기준으로 정리한다.
  - 필요 시 `scripts/`와 package scripts를 추가한다.
  - `.DS_Store`, `__MACOSX`, repository 문서, `node_modules`, source root 파일 제외를 검증 기준으로 고정한다.
- **Stage 3 — PR CI와 release runbook 작성**
  - 필요 시 `.github/workflows/ci.yml`을 추가한다.
  - `mydocs/manual/release_pipeline_guide.md`에 PR CI, release PR, tag/GitHub Release, Chrome Web Store 제출 전 순서를 작성한다.
  - #37 Dashboard guide와 privacy URL 기준을 연결한다.
- **Stage 4 — 통합 검증과 최종 보고**
  - CI/workflow syntax, package script dry-run, build/typecheck/test, grep, diff 검증을 수행한다.
  - 최종 보고서와 PR 본문에 release 전 남은 blocker를 분리한다.

## 검증 계획

### 단계별 검증

- Stage 1
  - `rg --files .github mydocs/manual mydocs/tech scripts package.json`
  - `rg -n "release|main|tag|Chrome Web Store|package|zip|CI|workflow|artifact|privacy" mydocs/manual mydocs/tech mydocs/report package.json .github`
  - 필요한 GitHub Actions 공식 문서 또는 action README 확인
- Stage 2
  - `npm run build`
  - package ZIP 생성/검증 dry-run
  - `unzip -l /tmp/crop-0.1.0-cws.zip`
  - `unzip -Z1 /tmp/crop-0.1.0-cws.zip`
  - `.DS_Store|__MACOSX|node_modules|mydocs|README|PRIVACY|NOTICE|THIRD_PARTY|LICENSE|package.json|vite.config|tsconfig` 제외 grep
  - `git diff --check`
- Stage 3
  - workflow YAML 구조 확인
  - `rg -n "npm run build|npm run typecheck|npm test|package:cws|verify:cws|devel|main|Chrome Web Store|PRIVACY.md" .github mydocs/manual mydocs/tech package.json`
  - `git diff --check`
- Stage 4
  - `npm run build`
  - `npm run typecheck`
  - `npm test`
  - package ZIP 생성/검증 dry-run
  - `git diff --check`
  - `git status --short`

### 통합 검증

- PR CI가 실행할 명령과 실패 기준이 문서와 workflow/package scripts에서 일치한다.
- 제출 ZIP 생성/검증 기준이 `.DS_Store`, `__MACOSX`, repository 문서, `node_modules` 제외를 보장한다.
- `devel -> main` release PR, tag/GitHub Release, Chrome Web Store Dashboard 제출 순서가 문서화된다.
- `main` 기준 privacy URL과 release/tag 기준 artifact 판단이 #37 산출물과 충돌하지 않는다.
- 실제 Chrome Web Store 제출과 실제 `main` 릴리즈는 수행하지 않고 후속 승인 항목으로 남긴다.
- `git status --short`가 PR 준비 전 빈 출력이다.
- `git diff --check`가 경고 없이 통과한다.

## 리스크

- **CI 범위 과다**: 처음부터 release artifact upload까지 CI에 넣으면 workflow 권한과 artifact 보관 정책이 복잡해질 수 있다. 기본 PR CI는 build/typecheck/test/package dry-run으로 제한하고, release artifact upload는 필요 시 별도 단계로 둔다.
- **공식 문서 루트 부재**: 사용자/기여자용 공식 문서 루트가 아직 없다. 이번 task는 maintainer 운영 절차이므로 `mydocs/manual`에 두고, 공개 contributor docs가 필요하면 별도 task로 분리한다.
- **GitHub Actions 버전 변동성**: `actions/checkout`, `actions/setup-node` 버전과 cache 정책은 바뀔 수 있다. Stage 1에서 primary source 기준으로 확인한다.
- **Store 제출과 release 순서 혼동**: Dashboard draft 입력과 `Submit for review` 시점이 섞이면 `main` privacy URL과 package artifact가 불일치할 수 있다. runbook에서 draft 가능/submit 금지 시점을 분리한다.
- **스크립트 추가 범위**: `scripts/`를 새로 만들면 유지보수 대상이 늘어난다. one-off 명령으로 충분한지 Stage 1/2에서 판단하고 구현계획서에서 확정한다.

- 실제 `main` 릴리즈 PR, GitHub Release/tag, Chrome Web Store 제출은 이번 task 범위에서 제외하고 후속 승인 항목으로 남기는 방향을 승인한다.
- 위 Stage 1~4 구성과 검증 계획으로 구현계획서 작성을 진행하는 것을 승인한다.
