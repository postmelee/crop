# Task #35 수행계획서

GitHub Issue: [#35](https://github.com/postmelee/crop/issues/35)
마일스톤: M020

## 목적

전체 페이지 캡처가 browser canvas 최대 dimension/area 제한을 넘는 긴 문서에서도 가능한 경우 단일 PNG preview, Copy, Save 경로까지 성공하도록 보정한다. 원본 해상도 그대로 하나의 canvas에 담을 수 없는 경우에는 종횡비를 유지한 상태로 출력 이미지만 자동 다운스케일한다.

이번 task는 `chrome.tabs.captureVisibleTab()`과 content script scroll stitching이라는 기존 권한/구조를 유지한다. 권한 확대나 native full-page screenshot API 전환 없이, 현재 stitching pipeline의 출력 크기 정책을 실패 중심에서 fallback 중심으로 바꾸는 것이 목적이다.

## 배경

작업지시자가 2026-06-04에 Chrome 확장 오류 화면을 공유했다. 오류는 `[crop] Failed to capture full page: Full page capture exceeds the maximum canvas size...`였고, 현재 구현의 `validateEstimatedOutputSize()`와 `validateOutputPixelSize()`가 page CSS size와 capture scale을 곱한 출력 크기를 `MAX_CAPTURE_DIMENSION`/`MAX_CAPTURE_AREA` 기준으로 차단하면서 발생한다.

Task #15에서 full page capture와 scroll stitching을 구현할 때 큰 canvas는 명시 오류로 처리했다. README도 이 제한을 문서화했지만, 실제 사용자 관점에서는 전체 페이지 기능이 큰 페이지에서 실패한다. 따라서 #35는 제한 초과 시 최대 canvas 안에 들어오는 scale을 계산해 단일 PNG로 fallback하는 품질 보정 작업이다.

## 범위

### 포함

- stitching 출력 pixel size 계산 정책 보정
- 제한 초과 시 종횡비 유지 다운스케일 fallback 적용
- full page capture의 oversized document 경로 회귀 테스트
- 선택 영역 stitching도 같은 canvas 제한 helper를 공유하므로 필요한 회귀 테스트 포함
- 출력 크기, 적용 scale, max dimension/area 준수 검증
- README와 Phase 6 품질 매트릭스의 제한 설명 갱신 필요 여부 판단 및 최소 반영

### 제외

- 원본 해상도 다중 PNG 분할 저장 UX
- Chrome `debugger` API 기반 screenshot
- `<all_urls>` 또는 broad host permission 추가
- cross-origin iframe full page stitching
- Firefox privileged `drawSnapshot()` 또는 browser-native screenshot API 포팅
- capture 중 lazy loading/layout shift/sticky 변화의 pixel-perfect 해결

## 설계 방향

- 현재 `MAX_CAPTURE_DIMENSION = 32767`, `MAX_CAPTURE_AREA = 268_435_456` 정책은 유지한다. 제한 자체를 높이지 않는다.
- 입력 tile capture는 기존처럼 viewport natural pixel 크기로 수집한다. fallback은 최종 stitching canvas와 destination rect에만 적용한다.
- 기본 크기가 제한 안에 있으면 기존 출력 scale과 pixel size를 그대로 유지한다.
- 기본 크기가 제한을 넘으면 `maxDimension / width`, `maxDimension / height`, `sqrt(maxArea / area)` 중 가장 작은 비율을 적용해 전체 output scale을 낮춘다.
- fallback scale은 x/y에 같은 비율을 곱해 종횡비를 유지한다.
- 너무 작거나 비정상인 입력은 기존처럼 명확한 오류로 처리한다. 다운스케일로도 유효한 canvas를 만들 수 없는 경우는 실패를 유지한다.
- `createFullPageTilePlan()` 단계에서 oversized output을 미리 throw하지 않도록 조정하고, 실제 captured image scale을 아는 stitching 단계에서 최종 fallback size를 결정한다.
- 선택 영역 stitching은 같은 `stitchCapturedTiles()` helper를 쓰므로 일반 크기 selected capture가 회귀하지 않도록 테스트를 함께 고정한다.

## 잠정 단계

- **Stage 1 — 출력 크기 정책과 다운스케일 helper 작성**
  - `stitch-image`의 output pixel size 계산을 제한 초과 fallback 가능 구조로 바꾼다.
  - 일반 크기, 한 변 제한 초과, area 제한 초과, fractional scale의 결과를 단위 테스트로 고정한다.

- **Stage 2 — full page/selected stitching 경로 통합**
  - `full-page-capture`의 oversized estimated output throw를 fallback 경로와 충돌하지 않도록 보정한다.
  - `crop-overlay`가 fallback 결과의 실제 outputWidth/outputHeight/tileCount를 기존 preview/action metadata에 유지하는지 확인한다.
  - full page와 selected page rect capture가 같은 helper를 안정적으로 공유하도록 회귀 테스트를 보강한다.

- **Stage 3 — 문서와 품질 매트릭스 갱신**
  - README 계열의 “큰 canvas 크기 오류” 설명을 “자동 다운스케일 fallback과 잔여 한계” 기준으로 갱신한다.
  - Phase 6 matrix에 oversized full page fallback 항목을 추가하고 수동 smoke 기준을 정리한다.

- **Stage 4 — 통합 검증과 최종 보고**
  - 전체 typecheck/test/build 또는 필요한 통합 검증을 실행한다.
  - 권한 경계 grep과 산출물 보고서를 정리한다.
  - PR 전 최종 상태와 잔여 리스크를 기록한다.

## 리스크

- **품질 저하**: fallback은 원본 해상도를 낮춘다. 출력 metadata와 문서에 다운스케일 가능성을 명확히 남긴다.
- **메모리/성능**: 제한 안으로 줄여도 큰 PNG 생성은 비용이 크다. 기존 max area를 유지해 browser crash 위험을 낮춘다.
- **scale rounding seam**: destination rect를 축소 scale로 다시 snap하면서 tile seam이 생길 수 있다. destination pixel edge alignment 테스트를 보강한다.
- **selected capture 회귀**: selected page rect도 같은 stitching helper를 사용한다. 일반 크기와 oversized selected rect를 테스트로 나눠 검증한다.
- **layout shift 한계**: lazy loading, animation, sticky layout 변화는 다운스케일과 별개로 남는 scroll stitching 한계다. 이번 task에서는 canvas 제한 fallback만 다룬다.
